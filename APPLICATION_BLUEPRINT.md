# BetterCMS — Application Blueprint

> Complete handoff specification. Read top to bottom to understand the product; every section is written so an engineer (or an AI coding agent) can rebuild the application exactly. The companion file `DESIGN_SYSTEM.md` covers all visual tokens and UI patterns; `WORKSPACE_AUDIT.md` and `AGENT_PLAN.md` hold deeper background on two subsystems.

---

## 1. What this is

BetterCMS ("Content Canvas") is a **multi-workspace, multi-project CMS** in the space of Webflow + Sanity + Prismic, with a first-class AI agent. It is currently a **frontend-complete v1 skeleton**: every screen, flow, store and guard exists and works in the browser; all data is in-memory or localStorage (seeded demo data); Supabase powers real email auth only. Production work = swap the module stores for a backend keeping the same action APIs.

Product pillars:
1. **Structured content, visual editing** — developers define sections/schemas in code; marketers compose pages from approved sections; editors edit inline.
2. **An agent that does the busywork** — plans first, stages drafts, human approves, one-click undo, fully audited.
3. **AI-answer-engine delivery** — every page also serves as markdown (llms.txt, .md twins, content negotiation).
4. **Enterprise controls** — roles that cascade, custom roles with scoped permissions, workspace AI governance.

Live demo: https://bcms-demo.vercel.app (guest door: "Continue without signing in"; noindex enforced at header/meta/robots.txt level).

## 2. Tech stack

| Layer | Choice |
|---|---|
| Framework | TanStack Start (file-based routes in `src/routes`, `routeTree.gen.ts` auto-generated by dev server) + React 19 |
| Build | Vite 7, Bun, Nitro (deploy preset auto-detects Vercel/Cloudflare; `NITRO_PRESET=vercel bun run build` produces `.vercel/output`) |
| Styling | Tailwind v4 (`@theme inline` tokens in `src/styles.css`), tw-animate-css |
| UI primitives | Radix (shadcn-style wrappers in `src/components/ui`), lucide-react icons, sonner toasts |
| Rich text | TipTap (starter kit + link/image/placeholder/typography/underline) |
| DnD / charts | dnd-kit, recharts |
| Auth | Supabase (email/password + reset). SSO buttons are simulated. Guest mode bypasses auth via localStorage flag |
| Dark mode | `.dark` class on `<html>` via `src/lib/cms/appearance.ts` (localStorage `bettercms.appearance.v2`: dark/light/system) |

**SSR rules learned the hard way**: never import `*.client.*` modules from server-reachable code (TanStack import-protection hard-fails the build); read `process.env` inside functions, not module scope; the app serves `X-Robots-Tag: noindex...` on every response via `src/server.ts` (demo privacy).

## 3. State architecture (the most important pattern)

One **central store** + many **module stores**, all with the same shape:

- **Central**: `src/lib/cms/store.ts` — workspaces, projects, collections, entries, domains, members, notifications, audit. Custom hook `useCMS(selector)` (NOT zustand) + non-React reader `getCMSState()`. Actions objects: `workspaceActions` (create/update/slugTaken/setWorkspacePlan), `projectActions.create`, `domainActions` (add/setStatus/setPrimary/remove), `entryActions`, `recordAudit`, `recordAgentAudit`.
- **Module stores** (`useSyncExternalStore`, keyed by projectId or workspaceId, lazy `ensure()` + `seed()`):
  - `cms/pages-store.ts` — PageDoc list per project (seeded 5-page marketing site: Home/About/Pricing(draft)/Blog/Contact(scheduled)). `usePages`, `getPages`, `pagesActions.add/update(patchFn)/remove/replace`, `buildPage`, `newPageId`. NOTE: `update` takes a patch FUNCTION `(p) => PageDoc`.
  - `cms/schema-store.ts` — visual model builder state (page/collection/block kinds, fields incl. reference + multireference, groups, section zone).
  - `cms/comments-store.ts` + `lib/comments/*` — field-anchored threads, reactions, mentions, read state.
  - `forms/forms.store.ts` — localStorage-backed (`bettercms.forms.v1`), full CRUD: forms, fields (12+ kinds incl. phone w/ country picker, business-email toggle), submissions (active/spam), integrations, after-submit config, Turnstile option. All access SSR-guarded.
  - `agent/*` — see §7.
  - `brand/brand-store.ts` — per-project BrandKit (colors 6 slots, heading/body fonts from `BRAND_FONTS`, radius, logos, voice tone/do/dont/protected) + `parseDesignMd` + `hasBrandVoice`; version bump per save.
  - `md/md-store.ts` + `md/serialize.ts` — see §9.
  - `workspace/custom-roles-store.ts` — see §6.
  - `workspace/tokens-store.ts` — API tokens + webhook endpoints per workspace (one-time reveal pattern).
  - `agent/governance-store.ts` — AI controls per workspace (§8).
  - `onboarding/onboarding-store.ts` — localStorage profile (role/usage/team/source/name).
  - `lib/guest.ts` — guest flag; `lib/cms/use-folders.ts` — dashboard folders (localStorage).
- **Persistence policy**: everything is in-memory except: workspace identity overrides, forms store, onboarding profile, guest flag, appearance, folders. Hard refresh reseeds the rest — expected demo behavior; production persists all stores server-side behind the same action signatures.

## 5. Identity, workspaces, plans

**Demo seats** (`workspace/my-role.ts` `MY_WORKSPACE_ROLE`): flowtrix=owner("Admin"), atlas=developer, northwind=marketer, pixelforge=editor, wayground=reviewer; unknown/created workspaces → owner.

**Two-layer billing** (`billing/pricing.ts`, the pricing brief is source of truth): workspace plan (free/company/agency/team/enterprise — seats, white-label) × per-site plan (free/basic/pro/team/enterprise — bandwidth, locales, AI credits, features via `siteHas`). AI tiers are SPEED-framed: **Lite / Balanced / Max** (never model names; BYOK models are the only exception). `AI_ACTIONS` credit costs per action; `tierAllowed(plan,tier,actionId)`: Lite everywhere; Balanced on Pro+ and on Basic only for `BASIC_BALANCED_ACTIONS` (schema, rewrite, seo-page); Max on Pro+. Usage displays never use red.

**Project kinds**: headless / managed("Cloud") / hybrid — chosen in the New Project wizard (multi-step; `projectActions.create` slug-uniquifies). Delivery mode drives which nav tab set renders and hosting availability (`cms/delivery.ts`).

## 6. Roles & permissions

**Built-in seats + cascade** (`my-role.ts`): rank owner(5)>developer(4)>marketer(3)>editor(2)>reviewer(1). **View-as** global store lets you preview any role at-or-below yours; the whole app reads the EFFECTIVE role (`useEffectiveRole(wsSlug)` hook, `effectiveRoleFor` for store guards). Switcher pill lives in the visual editor sub-header.

**Tab allow-lists** (`visibleTabs`): owner/dev → all; marketer → content, pages, collections, visual, agent, media, seo, analytics; editor → collections, visual, media, agent; reviewer → collections, visual. Helpers: `canSeeDeveloper` (dev+), `canCompose` (marketer+), `canEditContent` (all but reviewer), `canPublish` (marketer+). Reviewer's visual editor is forced to comment mode.

**Custom roles** (per project, `custom-roles-store.ts`, UI in Settings→Access→Roles tab): `{ base: marketer|editor|reviewer, capabilities: {edit, publish, seo, agent, generate, markdown}, scope: {collections, pages, sections}: "all"|string[] }`. Plan gates: custom roles on **Team+**; element/section-type scoping on **Enterprise only** (locked dashed row on Team; whole panel locked with "See plans" below Team). Seeded example "Blog author". Builder: name/description → base cards → 6 capability checkboxes → three ScopePickers ("All X" ↔ "Limit on" + 2-col checkbox list). ScopePicker onChange MUST use functional setState (same-batch toggle bug fixed). Runtime enforcement is v2/backend; layering rule: workspace AI controls beat role capabilities.

## 7. The Agent (flagship subsystem, `src/lib/agent/*` + `src/components/agent/*`)

**Trust model (non-negotiable product rules)**: the agent PLANS first → a person APPROVES → changes stage as DRAFTS (proposals) → apply through real store actions → **undo journal** enables one-click revert → everything audited. Nothing auto-publishes. Ever.

**Run lifecycle** (`runs-store.ts`, timer-simulated streaming): `planning → awaiting_approval → applying → review → done | rejected`. Read-only skills (audit/aeo/links) skip review and emit findings. Store-level guards: `roleCanAct` (editor+ via effective role), `clampTier` (plan, then governance ceiling), governance skill/generator/BYOK checks — all return `""` when blocked so no code path bypasses UI gating.

**Skills** (`skills.ts`, 6): draft (needs collection), backfill SEO metadata, audit (read-only), internal links (read-only), AEO readiness (read-only, Max/Pro+), migrate-from-URL. `skillFromPrompt` routes free text (aeo→migrate→links→audit→draft→backfill priority). `simulate.ts` builds plans (goal/items/boundaries incl. brand-voice line when `hasBrandVoice`), schema-aware proposals, findings from real store data; `applyProposals` re-validates live state and returns truthful appliedIds + `UndoOp[]` (`removeEntry`/`removePage`/`restorePageField`); `revertRun` only removes still-draft items and only restores unchanged fields.

**Surfaces**: `/p/$project/agent` page (hero composer, skills gallery with credit/gate hints, Generators section, `AgentRoster` named agents w/ schedules + Run now, `AgentHistory` w/ search/CSV export/per-run Undo, `ConnectedAgents` dev-gated) • `AgentDock` 400px in-flow sidebar on every project screen (toggle in header, hidden on /agent and for reviewers; `dock-store` open/show(runId)) • workspace `/agent` (requires @project chip; hands off via `?run=`).

**Composer** (`AgentComposer.tsx`): contentEditable with manual DOM chip spans (`data-kind/id/label`, inline SVG icons), `@` mention groups (pages/collections/entries/files/project-unlock at workspace scope), `/` skills, file attachments, tier dropdown + **BYOK** ("Your models" from `byok-store`, provider key dialog; BYOK runs bill to the key, credits=0; hidden entirely when governance disallows).

**External agents** (`connected-store.ts` + Connected agents surfaces): MCP endpoint + `npx bettercms connect` setup command, client list (Claude Code/Cursor/VS Code/custom), per-project scoped grants with one-time token reveal + revoke; same guardrails (staging-only writes, audited).

## 8. AI Governance (workspace Settings → AI controls)

`agent/governance-store.ts` per workspace: `{ monthlyCreditBudget: number|null, tierCeiling: AiTier, skills: Record<skillId,bool>, generators: {seo,abm}, byokAllowed, externalAgentsAllowed }`. ## 9. Pages, sections, visual editor, publishing

**Sections** (`components/cms/editor/sections/SectionSystem.tsx`): dev-defined `SECTION_DEFS` catalog (hero/logos/features/testimonial/cta/pricing/faq/contact...; variants, field defs, defaults, live render). Pages are ordered `SectionInstance[]`. `PAGE_TEMPLATES` (Landing/Case study/Event/White paper/Blog post) + custom save-as-template; `TemplatePicker` with live stacked previews; `instantiateTemplate`.

**Visual editor** (`visual.tsx`): dark-chrome around an always-light website canvas (`.bcms-preview-surface`); left Form panel (field inputs per section) ↔ Visual inline editing toggle; section controls (add between, swap variant, duplicate, delete — role-gated by `canCompose`); page switcher + Settings gear (PageSettingsDialog: name, path, seoTitle, seoDescription w/ 160-char counter, indexing); comments overlay; per-page **PublishMenu** (shared component, portal or inline): destination segmented Staging|Production, Publish now / Schedule (datetime-local), private preview link, Unpublish/Cancel schedule/Save as template/Archive. Page states: draft/published/modified/scheduled/archived (+`staged` flag). The generic header Publish button only shows on `/editor`; pages publish per-page.

**Pages hub** (`content.tsx`): compact table (title/path/sections, clickable status → PublishMenu, updated, ⋯ menu: open/settings/publish/Copy-as-Markdown/duplicate/delete), search + status SegmentedFilter (All/Published/Draft/Scheduled with counts), **Pages | Markdown** pill switcher, Generate menu + New page. `?batch=` shows `GenerationBatchBar` (title, live counts, Publish all, Undo via run journal, dismiss) with the table filtered to the batch.

**Comments** (`CommentSystem.tsx`): Figma/Sanity-style field-anchored threads, cross-mode pins (visual + form), reactions/mentions/images/reply/resolve, CSS-highlight text anchoring. Deliberately light-palette (floats over the light canvas).

**Rich text** (`RichTextToolbar.tsx`, reused across the CMS): inline bubble toolbar with portaled flyouts, H1–H6, AI panel, and a full link editor (internal page picker, new-tab toggle auto-applying noopener/noreferrer, rel as a single-select radio: None/nofollow/sponsored/ugc). Canvas links render underlined indigo. **TokenField** (SEO section): purple-chip editor for `{{field}}` CMS variables inside meta patterns.

## 10. Collections, schema builder, entries

## 11. Markdown delivery (AEO surface)

Position (research-grounded): *structured content serializes to markdown on request* — pages/entries are never stored as markdown; standalone .md files are the only markdown-first content.

`md/serialize.ts`: `pageToMarkdown` (frontmatter + per-section rules: hero→H1+CTAs bold, features→H2+bullets, faq→H3 pairs, testimonial→blockquote, cta/pricing/contact→H2+subtext, logos skipped; rich-text HTML stripped), `entryToMarkdown` (schema-driven: short fields bullet list, long fields H2 sections), `llmsTxt` (spec shape; published-only; root path → `/index.md`), `llmsFullTxt`.

`md/md-store.ts`: `{ llms:true, llmsFull:false, llmsMode:"auto"|"custom", llmsCustom, excluded[], files: MdFile[] (state draft|published) }` + `normalizeMdPath`. Seeded `/docs/getting-started.md`.

**Manager UI** (Pages hub → Markdown view): Delivery card — llms.txt row (Auto/Custom badge, copyable URL, Preview, **Edit** prefilled from generated → saving switches to Custom, **Upload**, Revert-to-auto), llms-full.txt (off by default + context-window honesty note), Content negotiation row (Always on + curl example). Endpoints table — every page/entry/file with source chip, per-row **Serve** switch (drops from llms.txt live), Preview dialog (mono, Copy/Download); files get StateBadge + Edit editor (**Save as draft / Publish**) + ⋯ (Publish/Unpublish/Copy/Download/Delete); uploads land as drafts; **drafts excluded from llms.txt/full**. Search + Type filter (All/Static/Collections/Files w/ counts). No plan gate (delivery is table stakes). Demo serves nothing at real URLs — serializers are the production contract.

## 12. Page generators (Prismic-parity)

Both are structured front-ends that create **agent runs** (`runsActions.startGenerator`) — history/audit/credits/undo free. Wizard = the approval, so runs go `applying → done`, pages land as DRAFTS with `batchId`+`generatedFor`.

**SEO pages from keywords** (Basic+, 12 credits/page, cap 50/batch): paste list or CSV (header-detected, extra columns become `{{tokens}}`) → template cards + URL prefix + title/description patterns w/ live Google-style example → review (count/credits/voice) → streamed steps → "Review pages" → batch bar. **ABM page** (Pro+, 60 credits): account + context textarea → template + section-by-section personalization plan → generates `/for/{slug}` draft with **noindex** → "Open in editor" opens `/visual?page=` with the **agent dock showing the run** for follow-ups. Entry points: Generate menu on Pages hub + Generators cards on the agent page.

## 13. Auth, onboarding, workspace creation

**/auth**: split screen (form left; brand panel right with product vignette + value props). SSO-first (Google/GitHub — simulated: toast + guest session; signup-mode SSO → /onboarding), real Supabase email/password (signup → stores name → /onboarding), forgot-password flow, **"Continue without signing in"** dashed button (the team demo door — keep verbatim), terms line.

Workspace step has live identity preview (avatar initial + `/w/{slug}`); invite step has email chip input; finish → `enableGuest()` + navigate to the new workspace (owner seat via fallback).

**/workspace/new** (from switcher's Create-workspace item): kind → name → invites, closable.

## 15. Cross-cutting product rules (enforce in any rebuild)

1. Agent: plan → human approval → drafts only → undo → audit. Generators inherit all of it.
2. Publishing is always explicit and per-item; exactly one status indicator per screen.
3. Tiers are speed names (Lite/Balanced/Max); never expose model names except user-owned BYOK.
4. Usage/credit displays never use red; gates explain themselves ("Off in AI controls", "Available on Enterprise") and link to plans.
5. Copy: no em/en dashes anywhere; no hype verbs (unlock/seamless/empower); calm, specific, sentence case.
6. Locked features render as explanatory locked panels, never hidden mysteriously; role-gating hides tabs instead.
7. Every destructive/irreversible action gets a confirm or an undo path; one-time secrets block dismissal until copied.
8. The website preview canvas is always light, even in dark chrome.
