Part 1

Welcome and product overview

This part explains what BetterCMS is, who it serves, how to open the demo, and the vocabulary you will meet on every screen. Read it first, with the demo open beside you, and every later part of this handbook will make sense.

How to use this handbook

This handbook is the design handoff for BetterCMS. It is written for designers, not engineers. It assumes you know Figma and product design well, and it explains every technical term the first time it appears.

The best way to read it is top to bottom with the demo open in a second window. Each part names the exact routes (the URL paths, like /w/flowtrix/p/flowtrix-co/content) and the exact UI copy you will see on screen, so you can click along as you read. When a part describes a flow, run the flow yourself. The demo is fully clickable and nothing you do in it can break anything, a hard reload resets almost all of it.

  • Parts 1 and 2 cover the product, the demo, and the navigation skeleton. Everything else hangs off these.
  • Middle parts go surface by surface: pages, content, editors, schema, SEO, agent, settings, billing, and so on.
  • Feature blocks inside each part follow one skeleton: what it is, why it exists, the user journey, the UI components, UX notes, visual notes, and future ideas. If you only skim, skim the feature blocks.

Read with the code nearby if you can. Every screen is one file in src/routes/, and the file name is the URL. For example w.$workspace.p.$project.seo.delivery.tsx is the screen at /w/{workspace}/p/{project}/seo/delivery. When this handbook cites a route, that is where the source lives.

Conventions used in this handbook

  • Anything in monospace is literal: exact UI copy, a route, a keyboard key, or a design token name. If the handbook writes Continue without signing in, that string appears on screen character for character.
  • Route placeholders use braces: /w/{workspace} means the workspace's slug goes there, so Flowtrix's dashboard is /w/flowtrix.
  • Status pills mark how built something is: Live means fully working in the demo, Simulated means the UI is real but the behaviour is staged, Preview means partially wired.
  • Role names are capitalized as the UI shows them (Admin, Developer, Marketer, Content editor, Reviewer). "Owner" and "Admin" refer to the same top seat; the interface label is Admin.
  • Screens are described at desktop size unless a note says otherwise. Tablet and phone differences are called out explicitly, following the device tiers described in Part 2.

What BetterCMS is

BetterCMS is an AI-first content management system (a CMS, the tool teams use to create and publish website content without redeploying code). It combines three products that normally live in three separate tools:

A visual site editor

Webflow-style page composition. Developers define sections (reusable page building blocks) in code, marketers arrange and edit them on a live canvas, editors change text inline. Layout stays safe because only developers can change structure.

A structured content model

Sanity-style schemas and collections. Content lives as typed fields (title, image, rich text), not as frozen page markup, so the same entry can feed the website, a mobile app, and an API at once.

A built-in AI agent

An operator that works inside the CMS with guardrails. It runs skills (predefined jobs like renaming a product across all content), proposes changes for human review, follows workspace instructions, and never publishes on its own.

The three halves meet through delivery modes, chosen per project when it is created:

  • Hosted (managed). BetterCMS renders and serves the site itself. The team gets the visual editor, hosting, domains, and the Publishing tab. Most marketing sites in the demo work this way.
  • Headless. BetterCMS holds the content and serves it over an API (a machine-readable feed), while the customer's own engineering team builds and hosts the frontend. The project keeps content, schema, SEO data, and forms, and gains a Hosting tab for building the customer frontend, but drops BetterCMS-side publishing.
  • Hybrid. A hosted site whose content also feeds other channels over the API. The demo's Flowtrix.co site is the example: one content model powers the website and a mobile app.

The demo you will use is a complete, clickable front end over in-browser data. Every screen, dialog, and state exists and responds, but there is no production server behind it. Part of the value of this handbook is telling you, surface by surface, which behaviours are real and which are simulated so you design against the intended product, not the demo's shortcuts.

The product's core beliefs

Four convictions shape every screen. When you design new surfaces, they are the tie-breakers.

Structure is separate from layout

Content lives in typed fields and collections; layout lives in developer-defined sections. A marketer can rearrange a page all day without ever being able to corrupt the data model, and a developer can restructure the model without touching the marketing site's look.

AI is an operator with guardrails

The agent acts like a careful colleague, not an autocomplete. It plans before it acts, shows every proposed change for review with a before and after, respects role and plan limits, obeys workspace budget ceilings, and cannot auto-publish. Trust is the feature.

Markdown-native delivery for AI answer engines

Every page can serialize to a markdown twin (a plain-text version of the page), plus llms.txt and llms-full.txt index files, so AI assistants like ChatGPT and Perplexity can read the site cleanly. Being the source AI answers cite is treated as a first-class distribution channel, like SEO was.

Calm plan gating

Locked features stay visible but quiet: a small plan pill, a short explanation, an upgrade path. No aggressive upsells, no dead ends, no surprise modals. A designer on the Free plan should always understand what a paid plan would add.

How to open the demo

The demo is live at bcms-demo.vercel.app. The sign-in screen offers real email authentication, but you do not need an account:

  1. Open bcms-demo.vercel.app. You land on /auth, a split-screen sign-in page.
  2. Below the sign-in form, use the guest door: the dashed button labelled Continue without signing in, captioned Explore the full demo with seeded workspaces. No account needed.
  3. You land in the Flowtrix workspace at /w/flowtrix, the default demo home.

To run it locally, the project uses bun (a JavaScript runtime, like Node) and serves on port 8080: run bun install then bun run dev in the repo root and open http://localhost:8080. The same guest door applies.

The auth screen is partly real. Email sign-up, sign-in, and password reset go through a real backend (Supabase). The simulated SSO buttons (Google, GitHub) and the guest door skip auth entirely and set a local guest flag. Signing out from the workspace switcher returns you to /auth.

Your first ten minutes

If you have never opened the demo, this short tour touches every major surface and makes the rest of the handbook concrete:

What is real and what is simulated

The demo's honesty model matters when you design: some flows exercise real infrastructure, most run on in-browser stores (JavaScript state kept in the browser tab), and a handful of things persist in localStorage (the browser's small local database that survives reloads).

LayerStatusDetails
Email auth (sign up, sign in, password reset)RealBacked by a real auth service. The confirmation email genuinely arrives.
All content, pages, entries, settings editsIn-browserClient-side stores seeded from mock data. Edits work instantly and fully, then reset on a hard reload.
Wizard-created projects and workspacesPersistedProjects made through the New project wizard, and workspace name, slug, and logo edits, are saved to localStorage and survive reloads.
Share links and cloneable templatesPersistedPublic /p/{token} sandbox links survive reloads on the same browser.
UI preferencesPersistedTheme choice, sidebar width and collapsed state, and similar small preferences persist locally.
Deploys, hosting, domain verificationSimulatedDeploy progress is an animation. Domain checks always succeed. Design the states as if they were real.
Presence, multiplayer cursorsSimulatedA local engine animates believable teammates from the real seeded member lists. No network multiplayer.
The AI agent's runsSimulatedPlans, streaming output, and proposed changes are scripted against real store data. The review and apply flow genuinely mutates the stores.
Third-party integrationsVisual mock-upsIntegration cards, webhooks, and API keys render and behave (including one-time key reveal) but call nothing external.

Hard reload is the reset button. If a demo ever looks broken or over-edited, a hard reload restores the seeded state. Only the persisted rows above survive. Mention this to anyone you hand the demo to.

Two practical consequences for design work. First, latency in the demo is zero because there is no network: real save, publish, and search operations will need loading, optimistic, and failure states that the demo never shows, so design them anyway. Second, the demo never errors: form validation exists, but server failures, conflicts, and permission errors are largely undesigned territory, and later parts flag where those states are missing.

Part 2

Information architecture and navigation

BetterCMS has one navigation skeleton: a workspace level and a project level, joined by a single persistent top bar. This part maps every surface, who sees it, and on which devices, so you always know where a new feature should live.

The two levels

Everything in the product sits at one of two altitudes.

Workspace level is where a team manages itself: the projects dashboard, the team roster, plans and billing, usage, domains, and workspace settings. It renders with a fixed left sidebar (252px) and a scrolling main area.

Project level is where the actual site work happens: pages, content entries, the visual editor, schema, SEO, forms, and project settings. Entering a project swaps the left sidebar for a horizontal tab row under the top bar, giving the work canvas the full width.

The content hierarchy

  • Account (you)
    • Workspace
      • Members and seats
      • Plans, billing, usage, domains
      • Project (site)
        • Pages
          • URL folders and organizer folders
          • Sections on each page
        • Collections
          • Entries
            • Fields and blocks
        • Schema, media, forms, SEO, settings

One account can belong to many workspaces; each workspace owns its projects; each project owns its pages and collections.

The rule of thumb: anything about people and money is workspace level, anything about one site's content and delivery is project level. The one deliberate overlap is Domains, which are owned by projects but also get a workspace-level roll-up page that groups them by project and routes into each.

The two levels also feel different on purpose. The workspace level is calm and administrative: a fixed sidebar, card-based pages, generous whitespace. The project level is a workbench: dense tab navigation, tool panels, and a canvas. The shared top bar is what keeps the switch from feeling like changing apps.

The top bar, end to end

Global top bar

A single 48px header (GlobalTopBar in src/components/cms/shell/GlobalTopBar.tsx) that persists across every signed-in screen. It is a three-zone grid: context on the left, search in the center, utilities on the right.

Why it exists

With two navigation levels and five roles, users need one stable anchor that always tells them where they are, lets them jump anywhere, and carries the app-wide controls (theme, notifications, role lens). Nothing else in the app is allowed to duplicate these.

User journey
  1. At workspace level, the left zone shows the BetterCMS logo, which links home. On phones a hamburger button (aria-label="Open navigation") appears beside it and opens the workspace drawer.
  2. Entering a project, the logo is replaced by the breadcrumb: workspace name, a separator, then the project name, followed by a tiny status dot (green published, grey draft, amber scheduled). The breadcrumb routes back up.
  3. The center is a 440px search pill reading Search or jump to… with a ⌘K keycap. Clicking it (or pressing Cmd+K) opens the command palette. On phones the pill collapses into a search icon on the right.
  4. The right cluster runs, in order: the view-as pill, the presence avatar stack, the theme toggle, a help button (? opens the shortcut cheatsheet), then inside a project a Connect button and a settings gear, and finally the notifications bell.
UI components
  • Breadcrumb (ProjectBreadcrumb) with the publish-status dot.
  • Search pill with keyboard hint, hover states that lift border and text color.
  • View-as pill (ViewAsControl): an eye icon plus the current effective role label, for example Admin, with a caret. Opens a 264px dropdown headed View as listing every role at or below yours, each with an icon, a one-line blurb, a (you) marker on your real role, and a check on the active one. While previewing a lower role the pill turns indigo so the lens is visible everywhere. It renders nothing outside a workspace or for reviewers (nothing below them to preview).
  • Presence avatar stack (TopBarPresence): overlapping avatars of teammates currently in the project, with a who-is-here popover and jump-to actions. Hidden on phones.
  • Theme toggle: light and dark switch.
  • Help button: opens the keyboard shortcut cheatsheet, also bound to the ? key.
  • Connect button (project only): plug icon, tooltip Connect your AI (Claude Code, Cursor, VS Code…). Opens the dialog for wiring external AI tools to the project over MCP.
  • Settings gear (project only): a shortcut into project settings.
  • Notifications bell: a badge shows the unread count, capped at 9+, ringed to sit cleanly on the bar. The dropdown is 320px wide with a header, the latest six notifications, and a View all notifications footer link to /w/{workspace}/settings/notifications. Empty state copy: You're all caught up.
UX notes
  • Cmd+K must work from anywhere, including inside editors, and the palette must open in under a frame. It is the primary navigation for power users.
  • The indigo view-as state is a safety feature: a user previewing as Reviewer must never wonder why buttons are missing. Do not restyle it to match the neutral pills.
  • On phones the bar drops to icon-only: presence, help, and Connect disappear, search becomes an icon, and the hamburger appears at workspace level only (projects on phones use the focused editor with its own navigation).
  • Every icon button is a 32px hit target with an aria-label, and the bell's label includes the unread count for screen readers.
Visual design notes
  • The bar uses the paper surface token (--topbar), a hairline bottom border, and 48px height (h-12). The project tab row below it is also 48px, giving a stable 96px chrome above project content.
  • Icons are 16px at 1.75 stroke; hover states use the shared row-hover token, focus uses the 2px ring token.
  • The status dot is 6px, color-coded to publish state; it carries a title tooltip since color alone is not enough.
Future improvements
  • An environment indicator (Local, Staging, Production) with a danger tint for production, called for in the design system notes but not yet built.
  • Breadcrumb dropdowns for sibling switching (click the project name to jump to another project without going home).
  • Notification grouping by project, and inline actions (approve, reply) inside the dropdown.

Where is the account menu? Not in the top bar. The personal account menu (UserMenu) sits at the bottom of the workspace sidebar, and workspace-wide actions (create workspace, invite members, sign out) live in the workspace switcher at the top of that same sidebar. The top bar right cluster is deliberately app-utilities only.

Workspace sidebar and settings navigation

At workspace level a 252px sidebar (Sidebar in src/components/cms/shell/Sidebar.tsx) provides all navigation. Top to bottom:

Command palette and global search

Command palette (Cmd+K)

A single overlay (CommandPalette) that is both the app's global search and its command runner, opened from the top bar pill or Cmd+K.

Why it exists

The product has over sixty routes across two levels. The palette collapses all of them, plus content itself, into one keyboard-first jump box so nobody has to remember where anything lives.

User journey
  1. Press Cmd+K anywhere. The input reads Type a command or search….
  2. With no query, grouped suggestions show contextual Actions, Create commands, Editor and Preview commands (inside a project), Navigate targets, and your Recent items.
  3. Typing filters across groups: Jump to… matches pages, collection entries, and components by name (with their tree path shown), Projects and Workspace match destinations, and Switch workspace always lists the other workspaces.
  4. Enter navigates or runs the command; the palette closes.
UI components
  • Search input with placeholder, grouped result list with section headings, keyboard-highlighted active row, empty state, and per-row icons and path captions.
UX notes
  • Recents make the palette feel personal; they come from a real recently-visited store per project.
  • Group order is intentional: actions before navigation, so a short query like "pub" hits Publish before a page named "Publications".
  • Full keyboard operation: arrows, Enter, Escape. The trigger pill advertises the shortcut.
Visual design notes
  • Standard modal surface tokens, uppercase group headings in muted text, generous 8px row padding for scanability.
Future improvements
  • Full-text content search from the palette (today content search lives in the dedicated search workspace), fuzzy matching, and AI "do this for me" commands that hand off to the agent.

Scroll, breadcrumb, and consistency conventions

These conventions hold across every surface. Breaking them anywhere makes the whole product feel less trustworthy.

  • One tree. There is exactly one hierarchy: account, workspace, project, content. No feature may invent a side hierarchy or a second "home". New features attach to an existing node.
  • One top bar. The global top bar is the only full-width header. Feature screens never add their own top-level headers, search boxes, or notification entry points.
  • Stable tab order. The project tab order never reshuffles per user. Roles and devices remove tabs, they never reorder the survivors, so muscle memory holds across seats.
  • Chrome is fixed, content scrolls. The top bar and project tab row never scroll away. Each main area manages its own scrolling (min-h-0 flex panes), and wide tables scroll inside their own container rather than the page.
  • Breadcrumbs go up, tabs go across. The breadcrumb is strictly ancestry (workspace, then project). Lateral movement is always tabs or sidebar items. Deep editors add their trail inside the content area, not in the top bar.
  • URLs are shareable. Any state worth returning to (active tab, selected node, view mode) lives in the URL path or query, so links pasted into chat land exactly where the sender was, subject to the recipient's role and device.
  • Persistent layout preferences. Sidebar width, collapsed panels, and theme survive reloads. If a user arranges the chrome, it stays arranged.
  • Escape hatches everywhere. Blocked states (wrong role, small screen, locked plan) always explain themselves and offer a path: the view-as pill shows what a role lens is doing, the LargerScreen interstitial names the feature and links back, and plan gates state the required plan.

Placement test for new features. Ask three questions: is it about people or money (workspace level), one site's content or delivery (project level, pick the closest existing tab), or the person themselves (account level)? If a feature seems to need a new top-level tab, it more likely belongs as a sub-page of Settings, SEO, or Content. The tab row is a budget, not a shelf.

Design system

Every token, type size, component convention, and copy rule that makes BetterCMS read as one product. This is the reference part of the handbook: when you design a new screen, everything on it should be assembled from the pieces documented here.

Design principles

Six principles govern every visual decision in the app. They are extracted from the live code (src/styles.css and src/components/ui/), not aspiration.

Depth through tone, not shadow

Surfaces stack by changing background tone plus a hairline border. Shadows are reserved for floating layers only (dropdowns, popovers, modals). A resting card never has a drop shadow.

Dense but readable

This is an operations tool. Body text is 12.5 to 13px, rows are compact, spacing is tight but consistent. Scanability beats whitespace.

One confident accent

The UI is grayscale. The BetterCMS pink appears only for the primary action, active navigation, focus, and selection. Never decoration.

Every pixel is a decision

Consistent padding, aligned baselines, deliberate hierarchy. If a screen looks whipped up, it is wrong.

Theme-aware by default

Everything works in light mode and the Graphite dark theme. Components reference CSS variables (named color tokens), never hard-coded hex values.

Accessible always

WCAG AA contrast, visible focus rings, full keyboard paths, accessible names on icon-only controls, 44px touch targets on interactive rows.

Color tokens

All colors live as CSS custom properties (named variables like --primary that components reference instead of raw hex) in src/styles.css. Light values sit on :root, dark values in a .dark block. The theme toggle swaps them app-wide in one place.

In dark mode the ladder is the Graphite system: neutral cool grays with no hue tint, starting from a dark graphite chrome (not black) and getting lighter as elevation rises. Dark never inverts the ladder and never uses shadows to signal depth.

Inputs and form controls

The shared field recipe

Text inputs, selects, and textareas share one recipe: h-8, full width, rounded-md, a default border, the input background token (paper in light, lifted s1 in dark), 12.5px text, 10px horizontal padding. On focus the border turns primary and a 2px primary-tinted ring appears (an 18 percent pink mix, matching --shadow-focus). There is no browser-default blue outline anywhere.

  • Search inputs add a leading 14px search icon and a trailing clear button that appears only when there is a query (see ListToolbar below).
  • Textareas resize vertically only, same border and focus behavior.
  • Selects are native <select> elements styled with appearance-none plus an absolutely positioned chevron icon, so they match text inputs exactly.
  • Mono inputs (API ids, keys, slugs) add font-mono text-[12px].
  • Labels are persistent, 11.5px medium, above the field, never a colon. A placeholder is only ever a format example ("name@company.com"), never the label.

Switch, checkbox, radio

The switch is 20px by 36px, a full-round track that is pink when on and --s4 gray when off, with a 16px white thumb sliding between the ends. Checkboxes and radios come from the Radix-based primitives and pick up the pink for their checked state. The important rule is the ToggleRow convention: a boolean setting is never a bare switch. It is a labeled row inside a bordered card, with the label plus a one-line hint on the left and the switch on the right, and multiple ToggleRows stack in one rounded container separated by hairlines.

Segmented controls

Single-select filters use a segmented control: a full-round or rounded-lg track on --s2 with a hairline border, where the active option is a raised card-colored chip with a soft shadow and the inactive options are muted text. Used for view switches (Table and Gallery), status filters, and plan period toggles.

TokenField, the variable chip editor

TokenField

A single or multi-line text editor where {{field}} variables render as purple chips instead of raw braces. It powers the SEO title and description templates on /w/{workspace}/p/{project}/seo/pages.

Why it exists

Meta titles for collection pages are templates ("{{name}} | Acme Blog"). Editing raw {{curly}} syntax is error-prone for marketers, so the field displays each variable as a labeled chip while the underlying value stays a plain template string. Nothing about the syntax changes, chips are only how it is displayed.

User journey
  1. An editor opens the SEO pages tab and selects a dynamic page (one generated from a collection).
  2. The meta title field shows normal text interleaved with purple chips like name.
  3. They type freely around chips; the caret never jumps because the editor only rebuilds when the value changes from outside.
  4. An "Add field" menu (plus icon, with a search box) inserts a new variable chip at the caret.
  5. Deleting a chip removes the whole variable in one keystroke.
UI components
  • A contenteditable field styled like the shared input recipe.
  • Purple chips: rounded, violet 10 percent fill, violet 30 percent border, small semibold uppercase or mono label, violet-600 text (violet-300/400 in dark).

  • The violet family is the reserved color for "dynamic value" across the app; character counters switch to a violet "Dynamic" label when a template is present because the final length depends on data.

The fill is emerald when healthy, amber from 80 percent of the limit ("approaching"), and sky blue when over the limit. Never red, in either theme. The value label pairs with the bar in tabular numerals with units.

Overlays and layering

Dialogs

Two dialog builds exist. The Radix-based ui/dialog.tsx: portaled, centered, max-w-lg, rounded-xl, solid --elevated-modal paper, --shadow-3, a bg-black/30 backdrop with a slight blur, and a 150ms fade-and-scale entrance. The hand-rolled portal dialogs used by larger flows (share project, markdown editors) pin to top-[6vh] to top-[8vh] with rounded-2xl and a max height around 84 to 88vh, so tall content scrolls inside the dialog instead of moving it. Both share the header (title, one-line subtitle, close X), body, and right-aligned footer actions. Escape closes, focus is trapped and restored.

Drawers and sheets

Quick record inspection slides a sheet in from the right on the --s2 surface with --shadow-3. The rule of thumb: drawers for a peek, full detail pages for work. Mobile navigation uses a left drawer behind a dark scrim.

Dropdown menus and popovers

Both sit on --s2 (light) or the elevated s3 (dark) with rounded-lg, a border, --shadow-3, 12.5px items, and destructive items in the red family. Menus animate with a 95-percent zoom plus fade.

Command palette

Cmd+K opens the cmdk-based palette on the --s4 surface: a search field, grouped fuzzy results across entities and quick actions, fully keyboard-driven. It is the model for any global search surface.

Z-index

All floating layers share z-50 and rely on portal mounting order rather than an escalating z-index arms race. Overlays and their content are siblings at the same level; sticky in-page chrome stays below 50. If a design needs something above a dialog, the answer is usually a second dialog step, not a higher z-index.

Iconography

All icons come from lucide-react, a single outline icon set. Two sizes cover nearly everything: h-3.5 w-3.5 (14px) inside buttons, rows, and inline text, and h-4 w-4 (16px) for standalone icon buttons and empty-state icon wells. Decorative or oversized icons are rare; when an icon needs to feel lighter (locks, info glyphs) it uses strokeWidth={1.75} instead of the 2px default. Icons never appear without an accessible name when they are the only content of a control.

Motion

Motion is deliberately minimal, this is an operations tool. The rules:

  • Hover states transition colors only (transition-colors, about 120ms). Nothing moves on hover.
  • Durations come from tokens: fast 120ms, base 150ms, drawer 180ms, modal 200ms, all on a standard ease (cubic-bezier(0.2, 0, 0, 1)). A second V4 set (instant 80ms up to slow 320ms) covers editor interactions like slide-overs and group expansion.
  • Dialogs fade and scale from 95 percent; dropdowns zoom and fade; skeletons shimmer. That is roughly the full animation inventory.
  • Animation is intentionally absent from: resting cards, page transitions, table sorting, chart re-renders on theme flips, and anything that would delay input.
  • Reduced motion is honored twice: the OS-level preference, and an explicit account preference that stamps data-reduce-motion on the root and forces all animation and transition durations to effectively zero app-wide.

Accessibility rules

  • WCAG AA contrast on all text and interactive elements. The token values are pre-tuned for it (dark foreground 14.6:1, dark muted 6.4:1), so using tokens is how a screen inherits compliance.
  • Visible keyboard focus everywhere. A global rule gives every focus-visible element a 2px ring in the ring color with 2px offset; fields use the primary-tinted 2px ring. Focus is never suppressed without a replacement.
  • 44px touch targets on interactive rows and mobile controls, even when the visual element is smaller.
  • Never color-only status. Every status pairs its hue with a dot, icon, or text label. A color-blind user must be able to read Published versus Draft from the label alone.
  • Icon-only controls carry title plus aria-label; toggles use role="switch" with aria-checked; menus, tabs, and listbox rails use their proper ARIA roles.
  • Dialogs trap focus, restore it on close, and close on Escape. Row actions revealed on hover remain reachable by keyboard and visible on touch.

What must stay consistent

  • Depth comes from surface tone plus hairline borders. Shadows only on floating layers. --shadow-card stays none.
  • Both themes lift lighter with elevation. The dark ladder starts at graphite #141416 and never inverts.
  • The pink appears only for the primary action, active nav, focus, and selection. One primary button per surface.
  • Usage and spend are never red. Emerald healthy, amber at 80 percent, sky over.
  • Status always pairs color with a dot, icon, or label. Never color alone.
  • Inter for UI, IBM Plex Mono for anything machine-flavored. The type scale table above is the whole scale.
  • Controls are h-8 by default, h-7 compact. Icons 14px in controls, 16px standalone.
  • The 4px spacing grid and the 6/8/10/12/16 radius scale. No invented values.
  • Lists use ListToolbar for search and filters, SegmentedFilter with counts, and Paginator with 50/100/200 sizes and a plain-hyphen range label.
  • Booleans are ToggleRows in bordered cards. Settings groups use SettingsSection with the header inside the card.
  • Focus is a visible 2px primary-tinted ring, everywhere, in both themes.
  • Sentence case, no dashes, verb-first buttons, past-tense object-first toasts, type-the-name confirms for high-risk deletes.
  • Components reference tokens (--card, --border, --status-live), never raw hex or raw Tailwind palette classes for themed surfaces.
  • Reduced motion (both the OS preference and the account setting) genuinely disables animation.

Known drift

Places where the code currently deviates from the system, found while auditing src/ for this handbook. None are user-visible breakages, but each is a trap for anyone using the code as a reference.

  • Two button scales. src/components/ui/button.tsx defines default: h-10 px-4 with 13px text, while the product convention (and nearly all CMS screens) is h-8 with 12.5px semibold. New screens copying the primitive's default come out oversized. The primitive's sm size is the real product default.
  • Two dialog geometries. The Radix ui/dialog.tsx centers vertically with rounded-xl; the hand-rolled portal dialogs (for example MarkdownManager.tsx, ShareProjectDialog.tsx) pin at top-[6vh] to top-[8vh] with rounded-2xl and their own scrim. Functionally fine, visually two families. Pick the top-pinned geometry as canonical for tall content.
  • Raw slate and gray classes outside the preview. About two dozen files still use slate-* or gray-* Tailwind classes instead of tokens, for example the mobile nav scrim in AppShell.tsx (bg-slate-900/45) and role tint maps in workspace/RoleCard.tsx. Many hits inside editor/sections/ and editor/preview/ are legitimate (they render the light-locked website preview), but chrome-side uses should migrate to tokens. The dark remap layer only rescues status text hues, not slate fills.
  • Two tree indent formulas. The content page tree indents at 16 + depth * 22, the sidebar ContentTree at 12 + depth * 14. Defensible (different densities) but worth documenting as intentional or unifying.
  • Duplicate motion token sets. --motion-fast/base/drawer/modal on the theme roots and a V4 --motion-duration-* set in the Tailwind theme block coexist, with slightly different values. Consolidate before the production rebuild.
  • Dark motion easing differs. The .dark block redefines the motion curves with a bouncier bezier (0.2, 0.8, 0.2, 1) than light mode's standard ease. Almost certainly unintentional; themes should not change how things move.

Use the tokens and you inherit the system. Almost every rule in this part (contrast, theming, density, focus) is encoded in the CSS variables and shared components. The fastest way to design a compliant screen is to compose from ListToolbar, SegmentedFilter, Paginator, SettingsSection, ToggleRow, EmptyState, and the ui primitives, and to name colors only by token.

Part 4

Dashboard, workspaces and projects

This part covers everything a person sees between signing in and opening an editor: the workspace dashboard where projects live, the switcher between workspaces, the wizard that creates a project, and the flows that move projects around (transfers, share links and clones). It also covers the notification bell that sits above all of it.

The mental model: workspace, then project

BetterCMS has two levels of container. A workspace holds people, billing and a set of projects (think of it as the company or agency account). A project is one website or one headless content source. Every URL in the app reflects this: the dashboard is /w/{workspace} and a project lives at /w/{workspace}/p/{project}.

Container hierarchy

  • Account (you)
    • Workspace
      • Folder (dashboard only)
        • Project
      • Project (unfiled)
      • Members, billing, settings

Folders are a dashboard organizing layer only. They do not affect URLs, permissions or billing.

The demo ships with six seeded workspaces so every plan tier has a home: flowtrix (Agency), aarav (Free), northwind (Company), pixelforge (Agency), wayground (Team) and atlas (Enterprise). Guest sessions land in flowtrix by default.

Workspace dashboard

Projects explorer

The dashboard at /w/{workspace} is a deliberately minimal, projects-first screen: a workspace header with one primary New project button, then a single explorer where folders and projects sit side by side as rows (list view) or cards (grid view).

Why it exists

Most CMS dashboards bury projects under charts and announcements. Agencies and multi-site teams open this screen dozens of times a day just to jump into a site, so the design goal was: find and open a project in under two seconds, and keep large portfolios organized. The folder layer exists specifically for agencies that hold ten or more client sites in one workspace.

User journey
  1. The user lands on /w/{workspace} after sign-in or after picking a workspace in the switcher.
  2. A brief skeleton shimmer (about 220 ms) plays while the explorer mounts, then folders appear first, followed by unfiled projects.
  3. They either click a project row/card to open /w/{workspace}/p/{project}, click a folder to drill in (a breadcrumb appears: All projects › folder name), or type in the search field.
  4. Search is Webflow-style: it flattens across every project in the workspace and ignores folder boundaries, so a project is always findable regardless of where it is filed.
  5. To organize, they click the folder-plus icon to create a folder (New folder dialog, placeholder e.g. Marketing sites), then use each project's overflow menu, Move to folder, to file projects. A toast confirms every action.
UI components
  • Workspace header: an uppercase Workspace eyebrow, the workspace name as an h1, the line All your projects, organized. and the primary New project button with a pink glow shadow.
  • Toolbar: search input (Search all projects…), a sort dropdown (Last updated, Name, Status), a square New folder icon button, and a segmented list/grid view toggle.
  • Project row (list view): a seeded SVG site thumbnail, project name with a small framework mark (Next.js or Astro), the domain in muted text, a Type cell (Site, Headless or Hybrid with a tooltip explaining each), a status dot badge (Live green, Draft amber, Scheduled sky, Archived rose), a relative Last updated time, a site plan badge, and hover-revealed actions (an Open icon button plus the overflow menu).
  • Project card (grid view): a 16:10 thumbnail with the framework chip overlaid top-left, then name, domain, plan badge and status badge.
  • Folder row/card: a folder icon tile, name, project count, and in grid view a 2x2 collage built from the thumbnails of the projects inside (empty folders show a large muted folder glyph).
  • Project overflow menu: Open, Share & preview link, Duplicate, Move to folder (submenu listing folders with a check on the current one, plus Remove from folder), Transfer project (or Cancel transfer when one is pending), Archive, Delete.
  • Folder menu: Rename, Delete folder. Deleting moves the contained projects back to the root and says so in the toast.
  • Empty states: a first-run card (No projects yet with a New project button), a no-search-results state with a Clear search button, and an empty-folder state that explains how to move projects in.
  • Skeletons: shimmer rows or shimmer cards matching the active view.
UX notes
  • Whole rows and cards are click targets (with role="link" and Enter-key support), not just the title. The hover state shows a 2 px pink accent bar on the row's left edge.
  • Row action buttons are hidden until hover on desktop but always visible below the md breakpoint, so touch users are never stuck.
  • On phones the list collapses to Name + Status + actions; Type, Last updated and Plan return at md and up.
  • Folder navigation is one level deep by design (folder → projects). There are no nested dashboard folders, which keeps the breadcrumb to a single hop.
  • Overflow menus stop click propagation through React portals; without that, choosing a menu item would also trigger the row's navigation. Any redesign of these menus must preserve that behavior.

Known gap (design debt). In the project overflow menu, Open, Archive and Delete have no handlers wired: clicking them closes the menu and nothing happens. The hover-revealed Open icon button on rows is also inert. This came out of the engineering audit and is a real gap, not a stylistic choice. When these get wired, Delete (and arguably Archive) must gain a confirmation step; the audit flagged the app-wide lack of destructive-action confirms as a companion issue.

What is simulated. Thumbnails are deterministic SVG mockups seeded from the project id, not real site screenshots. The status column is only partly real: Live reflects the project's publish state, but Draft and Scheduled are rotated onto unpublished projects by row index for visual variety. Folders and their assignments persist in localStorage per workspace (key bettercms.folders.v1.{workspace}), so they survive reloads in the same browser but do not sync across devices or users.

Visual design notes
  • The page is capped at 1240 px and generously padded; the explorer itself is borderless rows on the canvas rather than a boxed table, which keeps the screen airy.
  • Status is a 6 px dot plus an 11.5 px label, never a filled pill; the Live dot carries a soft green glow. Plan badges are the shared SitePlanBadge component and must not be restyled locally.
  • List column header is sticky with a blurred translucent background so it holds up over long portfolios.
  • Cards lift 1 px and strengthen their border on hover; rows only tint their background. Focus is a 2 px inset primary ring on both.
  • Skeleton shimmer uses a 1.4 s translating gradient; keep loading under a beat, the delay exists only to prevent a flash of empty content.
Future improvements
  • Wire Open, Archive and Delete, with an archive section or filter and an undoable delete (trash with retention) rather than a hard confirm alone.
  • Drag-and-drop projects into folders, and multi-select with a bulk move bar.
  • Real site screenshots (or a live iframe snapshot) replacing the seeded SVG thumbnails once projects have stable URLs.
  • Pinned or favorite projects that float to the top across folders.
  • Per-project quick stats on hover (last deploy, open comments, pending workflow items).

Workspace switcher

Workspace identity and switcher

The block at the top of the left sidebar shows the current workspace (logo tile, name, and a subtitle like Agency workspace · Admin) and opens a dropdown for switching between all workspaces the person belongs to.

Why it exists

Freelancers and agency staff live in several workspaces at once (their own, plus one per client relationship). The switcher makes the current context unmistakable, shows the plan and your role in every workspace before you jump, and is the single entry point for creating a new workspace.

User journey
  1. The user clicks the workspace block in the sidebar (the chevrons-up-down icon signals it is a switcher).
  2. The dropdown opens with a Current workspace section at the top: logo, name, slug and a pink check.
  3. Below, a Switch to list shows every other workspace with an initial tile, the person's role in that workspace as a small outlined chip, and the workspace plan badge. With the six demo workspaces this list shows five entries.
  4. When someone belongs to more than five workspaces, an autofocused Find workspace… search field appears above the list and filters by name or slug.
  5. Picking a workspace navigates to that workspace's dashboard. The footer offers Create workspace (goes to /workspace/new), Workspace settings, Invite members, and a destructive-styled Sign out.
UI components
  • Trigger button: 32 px logo tile (uploaded logo or the name's first letter), two-line label, chevron affordance, hover row tint.
  • Dropdown (260 px): section labels in 10 px uppercase, scrollable workspace list capped at 264 px, role chips, WorkspacePlanBadge per row, an empty-search state (No workspaces match.), separators, and the four footer actions with icons.
UX notes
  • Role and plan are visible per row before switching, which matters when the same person is an Admin in one workspace and a reviewer in another; expectations are set before the jump.
  • The search field only appears past five workspaces, keeping the common case to one click and zero typing.
  • Sign out lives here as well as in the account menu; both clear the session and return to /auth.
  • Switching is a plain navigation, so the browser back button returns to the previous workspace, which people rely on.
Visual design notes
  • The current workspace is confirmed with a check, not a highlight, so the list stays quiet.
  • Plan badges keep their shared colors per tier; do not invent local variants. Role chips are neutral outlined text at 9.5 px.
  • The trigger's chevron brightens on hover and while open (data-state=open), a small but load-bearing affordance.
Future improvements
  • Keyboard palette entry (for example Cmd+O) to jump workspaces without the mouse.
  • Unread or attention counts per workspace row (pending transfers, review requests).
  • Recent-first ordering instead of seed order.

New project wizard

New project wizard

A multi-step dialog, opened from the dashboard's New project button, that forks on the first question: build headless (model content here, your own code fetches it over an API) or on BetterCMS Cloud (we build, host and deploy the site, labeled Managed).

Why it exists

These two audiences want opposite things from minute one. Developers picking headless need an API endpoint and a token; marketers or founders picking Cloud need their code (or a template) connected and a live URL. One wizard that forks early serves both without burying either in the other's questions.

User journey
  1. Step 1, type. How do you want to build? Two large cards: Headless CMS (bullets: Content Delivery API + SDKs, any framework, you keep your own hosting) and BetterCMS Cloud, visually highlighted with a Managed badge (bullets: hosting + continuous deploys, visual editor and one-click publish, technical SEO handled). Clicking a card advances immediately.
  2. Step 2, source (Cloud only). Bring your site: Connect GitHub (tagged Recommended, every push auto-deploys), Upload a ZIP (up to 100 MB), or Start from a template (four starters: Astro Blog, Next.js Marketing, Next.js Commerce, Astro Docs). Headless projects skip straight to step 3.
  3. Step 3, configure. The dialog narrows and shows source-specific UI: a GitHub connect card that flips to a repo picker (Connected as @acme), a dashed ZIP drop zone, or the template grid. Below: the Project name field (auto-filled from the chosen repo, ZIP name or template) and a framework chip picker (Next.js, Astro, Nuxt, SvelteKit, Other), hidden when a template already decides it. The primary button reads Create & deploy for Cloud, Create project for headless.
  4. Step 4, provisioning. An animated checklist ticks through stage lines: Cloud runs Creating project, then the source step (Linking repository / Uploading files / Cloning template), Provisioning cloud hosting, Running first build; headless runs Creating project, Generating content API, Issuing API keys.
  5. Step 5, done. Cloud shows Your site is live with the {slug}.bettercms.site URL, copy and visit buttons, and Open editor. Headless shows Your project is ready with the API endpoint, a masked project token behind a Reveal toggle, a copyable fetch snippet tailored to the chosen framework, and Open schema editor.
UI components
  • Dialog with a step eyebrow (New project, then Headless CMS or BetterCMS Cloud) and an inline Back button on middle steps; the dialog widens to 720 px for the card steps and narrows to 560 px for configuration.
  • Type cards with icon tiles, benefit checklists and a hover-revealed Continue hint; source rows with chevrons; template cards with framework marks; framework chip group; progress checklist with spinner, checks and pending dots; secret row with Reveal/Hide; code snippet block; copy buttons with toasts.
UX notes
  • Choice steps advance on click with no Next button, which keeps momentum; only the configure step has an explicit primary action, and it stays disabled until the name and source requirements are met.
  • Name auto-fill from repo, ZIP or template removes the most commonly skipped field. Users can still override it.
  • State fully resets every time the wizard opens, so an abandoned run never leaks into the next one.
  • The done step hands each audience their exact next action: a live URL for Cloud, a working code snippet for headless. Nothing generic.

What is simulated. The whole provisioning sequence is an animation on a timer (about half a second per line); no build or deploy runs. GitHub connect flips a local flag and shows four hard-coded repos, the ZIP drop zone fakes a file named site-export.zip, and the token and endpoint on the done step are derived strings, not live credentials. The project itself is real within the demo: it is created in the client store and opens in the editor.

Visual design notes
  • The Cloud card is the visually preferred path (primary-tinted border and icon tile plus the Managed badge); headless is equal in size but quieter. Keep that hierarchy deliberate, it encodes the business preference.
  • Provisioning lines turn from muted to foreground as they activate, with emerald checks on completion; the rhythm sells the "work being done" moment.
  • Secrets are masked with bullet characters by default; Reveal is a text button, not an icon, for clarity.
Future improvements
  • A real template gallery with live previews (the shared preview sandbox from share links could power this).
  • Import from an existing CMS (WordPress, Contentful) as a fourth source.
  • A slug and domain preview on the configure step, editable before creation.
  • Team defaults: remember the last framework and source per workspace.

Project transfers

Transfer a project

Move a project to a different workspace, modeled on Webflow's transfer flow. Two destinations: one of your own workspaces (instant) or someone else's email (a pending request they accept from their dashboard). Opened from the project overflow menu, Transfer project.

Why it exists

Agencies hand finished sites over to clients, and freelancers graduate personal projects into company workspaces. Without a transfer flow, handover means rebuilding or sharing credentials. The flow also enforces the billing rule: the destination workspace owns billing, so the site plan resets on transfer.

User journey
  1. The sender opens the dialog and picks a destination type with a two-card toggle: My workspaces (Move it instantly) or Someone else (They accept by email).
  2. Own workspace: a list of their other workspaces shows each name, plan and project count; picking one and confirming with Transfer project moves it immediately, with a toast like Flowtrix.co moved to Pixelforge.
  3. Email: they enter the recipient's email and an optional note (Handing this site over as agreed.), then Send transfer request. The request stays pending; the sender's overflow menu now shows Cancel transfer ({email}) instead of Transfer.
  4. Both paths show a persistent amber warning first: pages, content, schemas, forms and media all move; the site plan resets to Starter; custom domains disconnect; the new workspace owns billing.
  5. The recipient sees an accept banner at the top of their workspace dashboard: a primary-tinted card reading {Sender workspace} wants to transfer "{Project}" to you, with the target email, the note, and the line Accepting moves it into {this workspace} with its content, pages and media. Buttons: Decline (ghost) and Accept into {workspace} (primary).
  6. Accepting pulls the project into whichever workspace's dashboard they pressed the button on; that is how the recipient "picks" the receiving workspace.

Email transfer: the accept loop

Sender

Overflow menu→Transfer dialog→Send request→Pending (can cancel)

Recipient

Dashboard banner→Accept into workspace→Project appears, plan resets to Starter

Decline and cancel both end the request; the project never moves without an explicit accept.

UI components
  • Transfer dialog: header with the project name and source workspace, destination-type card toggle, workspace list with selection checks, email and note fields with validation (the button disables until a workspace is picked or the email parses), the amber consequences panel, Cancel and a mode-specific primary button.
  • Dashboard accept banner: transfer icon tile, sender and project line, meta line, Decline and Accept buttons; multiple pending offers stack.
  • Overflow menu state swap: Transfer project becomes Cancel transfer (email) while pending.
  • Toasts on every outcome: sent, moved, accepted ({Project} joined {Workspace}), declined, canceled.
UX notes
  • The consequences warning is always visible, not behind a second confirm step; the single primary click is the confirmation. If real billing gets attached, consider a type-to-confirm step for the email path.
  • The empty state for people with only one workspace (No other workspaces yet. Create one from the workspace switcher.) routes them to the fix rather than dead-ending.
  • Because the demo is single-player, "the recipient" is you opening any other workspace; the banner intentionally appears on every dashboard except the sender's so the loop can be demonstrated.

What is simulated. No email is sent, and pending transfer requests are held in memory only, so they disappear on a hard reload (unlike folders and share links, which persist in localStorage). The move itself is real within the demo store, and the plan reset and banner flow behave as specified. Production would email a signed accept link and run the move server-side.

Visual design notes
  • The banner uses primary-tinted border and background (color-mix at low opacity) so it reads as "action needed" without alarm red.
  • The warning panel is amber with a triangle icon, consistent with the app's other consequence notices; dark mode uses translucent amber.
  • Selected destination cards and rows use the primary-tinted selection treatment shared with the wizard's pickers.
Future improvements
  • A transfer history entry in the audit log, and a notification (not just a banner) for incoming offers.
  • Expiring requests with a visible countdown, and a resend action.
  • Enterprise: transfer with plan preserved when both workspaces share a billing relationship.

Share links and the public sandbox

Share links, read-only sandbox and cloneable templates

The Share dialog (reached from the project header and from the dashboard overflow menu item Share & preview link) mints two kinds of public link: a read-only preview and a template link. Both open the same public sandbox at /p/{token}; the template variant adds a Clone this project button that can be switched off.

Why it exists

Teams need to show a build to a client or stakeholder who has no account, without granting editing or exposing settings. Template links go further: they let a finished project be offered as a starting point that anyone can copy into their own workspace, or as look-only inspiration when cloning is off.

User journey
  1. The owner opens Share project and sees two toggle cards: Read-only preview link (Anyone with the link can review the build: pages, content model and entries. No editing, no settings.) and Template link.
  2. Flipping a switch mints the link and reveals a URL row with copy and open-in-new-tab buttons, plus a live view counter. The template card additionally reveals an Allow cloning switch whose helper text flips between The Clone button is shown to visitors. and Preview only. The Clone button is hidden.
  3. A visitor opening /p/{token} gets a standalone page outside the app shell: a top bar with the BetterCMS logo, the project name and a lock-icon pill (Read-only preview or Template), then three tabs: Pages (real rendered pages with a page list sidebar and a desktop/mobile device toggle), Content model, and Collections (entry tables with status dots).
  4. There is no sign-in, no editing, no settings anywhere in the sandbox. A footer line reads Shared read-only preview · Built with BetterCMS.
  5. On a cloneable template link, Clone this project opens a dialog to pick a destination workspace and name the copy (defaulting to {Project} copy). Cloning copies the project, drops the visitor into the app, and navigates to the new copy's content view.
  6. Turning a link's switch off kills it: visitors then see the invalid-link screen (This link isn't active with advice to ask the owner for a fresh link).
UI components
  • Share dialog: two ShareCard blocks with icon tiles, switches, monospace URL rows, copy-with-check feedback, view counters, the clone sub-switch, and a footer note: Links are unlisted and not indexed by search engines.
  • Sandbox: sticky header, tab bar with an active underline, pages canvas with device toggle, model and collection views, invalid-link screen, clone dialog with a workspace picker.
UX notes
  • The sandbox is a top-level route, deliberately outside the workspace auth gate; it must never grow navigation into the app shell beyond the clone flow.
  • Copy feedback is doubled (icon swaps to a check and a toast fires) because the URL row is the whole point of the dialog.
  • The link resolves client-side only; a brief spinner shows while the token decodes, which avoids a flash of the 404-style screen.

What is simulated, and one real security note. The token is self-encoded: it packs the project id and link kind into the URL itself rather than being issued and checked by a server. That is why links to seeded projects work in any browser, while a link to a freshly created project only resolves in the browser that created it. It also means turning a link "off" only takes effect for viewers because the sandbox checks local state; the engineering audit flagged that production must replace this with server-issued, revocable tokens. View counts only increment in the owner's own browser.

Pages, collections and the content model

Where content lives and how it gets made: the Pages hub with its folder tree, the collection workspace, the Notion-style entry editor, the schema builder that defines what an entry even is, and the Markdown manager that serves everything as plain text for AI agents. The mental model: three layers

Before the individual screens, hold one picture in mind. Content has three layers, each with its own home:

Schemas

The blueprint. A schema (also called a model) defines what fields an entry has, like "a blog post has a title, a cover image, and a body". Built by developers at /w/{ws}/p/{project}/schema.

Collections and entries

The structured content itself. A collection is a set of repeatable entries that all follow one schema (all blog posts, all authors). Edited in the workspace at /w/{ws}/p/{project}/editor.

Pages

The site. Static pages are built from visual sections; collection pages are rendered from entries. Both live in one tree on the Pages hub at /w/{ws}/p/{project}/content.

Everything here reads and writes in-browser data stores (src/lib/cms/pages-store.ts, folders-store.ts, store.ts, md-store.ts). The interactions are fully real, but most data resets on a hard reload. Where a specific feature is simulated, it is called out below.

The folder and URL model

Folders are the piece designers most need to internalize, because two visually similar things behave differently. A folder's kind is distinguished only by whether it carries a slug (the short lowercase name that appears in the URL, like solutions):

URL folder

Has a slug. It prefixes the URLs of pages placed inside it: a page with slug enterprise inside the Solutions folder serves at /solutions/enterprise. Nested URL folders chain their slugs.

Organizer folder

Has no slug. It is pure visual grouping in the Pages hub and has zero effect on any URL. It exists so teams can tidy a long list without touching the live site structure. Organizer rows carry a small organizer badge.

Folders nest up to four levels deep (MAX_FOLDER_DEPTH = 4 in src/lib/cms/folders-store.ts). Every project is seeded with two example URL folders so the feature never starts empty: Solutions for marketing pages, and Resources, which a demo collection is placed into so nested collection URLs are visible out of the box.

Collections are first-class citizens of the same tree. A collection in a URL folder composes its dynamic route from the folder prefix plus its own slug: the Posts collection inside Resources serves at /resources/posts/:slug (the :slug part means "each entry's own slug goes here"). A collection in an organizer folder, or in none, keeps its short /posts/:slug route. Moving a collection into a URL folder spells out the new route in the toast: Moved to Resources · /resources/posts/:slug.

How folders shape URLs

  • Pages root
    • Solutions (URL folder, /solutions)
      • Page "Enterprise" → serves at /solutions/enterprise
      • Nested URL folder /industries → pages serve at /solutions/industries/...
    • Resources (URL folder, /resources)
      • Posts collection → entries serve at /resources/posts/:slug
    • Team docs (organizer, no slug)
      • Page "Pricing" → still serves at /pricing
    • Loose page "/about" → serves at /about

URL folders prefix child URLs; organizer folders group without touching URLs. Maximum nesting is 4 levels.

The display-only move contract. Moving an existing page into a folder changes where it sits in the tree and what URL the hub displays, but it does not rewrite the page's stored path. The row shows a composed "effective URL" (folder prefix plus the last segment of the stored path) while page.path stays what it was. This is deliberate: silently rewriting live URLs on a drag would break links and search rankings. The prefix is only baked into the real path when a page is created inside the folder. Treat this as a contract to make visible, not a bug to hide; a future "also update the URL, with a redirect" affordance is the long-term answer.

Drag and drop

Reorganizing is direct manipulation, with a 5px activation distance so clicks never misfire as drags:

  • A grip handle (vertical dots) fades in at the left edge of rows on hover; dragging starts from the grip.
  • A ghost overlay follows the pointer: a small shadowed chip with an icon and the item's name, so the original row never visually leaves the table.
  • Drop a page onto a folder row to move it in; drop a folder onto another folder to nest it. The hovered target tints with 10 percent primary and gains an inset ring.
  • While anything is dragging, a strip appears under the table header reading Drop here to move to the top level, so escaping a folder is always one visible target away.
  • Illegal targets: a folder's own subtree is disabled as a drop target while it drags, and a drop that would exceed the four-level cap (counting the dragged folder's own subtree height) is rejected with a toast: Can't nest there. That would go too deep.
  • Drag changes only membership and nesting; row order within a level still follows the sort dropdown. Drag is disabled for read-only roles and in batch review.

Section-based page composition

Sections and the section library

Pages are ordered lists of sections, the same model as Prismic's "slices". Developers register each section in code (a name, category, one or more layout variants, a small field schema, defaults, and a real React renderer). Marketers then compose pages from that catalog without ever seeing the schema.

Why it exists

This is the core contract of the product: developers guarantee that anything a marketer can build is on brand, and marketers get full independence within that boundary. The demo ships eight sections (Hero, Feature grid, Logo cloud, Testimonial, Call to action, Pricing, FAQ, Contact) across four categories (Hero, Content, Social proof, Conversion), with ten layout variants in total.

User journey
  1. A composer-capable user hovers between two sections on the canvas. A thin indigo line with an Add section chip fades in at the insertion point. An empty page shows a dashed placeholder with a Browse sections button instead.
  2. Clicking opens the section library, a large modal titled Add a section with the subtitle 8 sections, 10 layouts. Defined by your developers, always on brand.
  3. They browse by category rail or search, preview each section as a live, scaled-down render of the actual component (not a screenshot), pick a layout variant chip if the section has several, and click the card. An Add section pill appears over the preview on hover.
  4. The new section is inserted at the hovered position, the canvas smooth-scrolls it into view, it flashes with an indigo ring for about 1.4 seconds, and a toast confirms, for example Hero added.
  5. Hovering any section on the canvas reveals its composition chrome: a name chip on the left (icon, name, current variant) and a control toolbar on the right with Change layout, Move up, Move down, Duplicate section and Delete section.
  6. New pages can also start from page templates (Landing page, Case study, Event page, White paper, Blog post), and any page can be saved as a custom template from the publish menu (Save as template).
UI components
  • Insert points: hover-revealed line plus pill button between every pair of sections, and a full-width dashed Add section button after the last one.
  • Section library modal: header with icon, title, count subtitle and search field; left category rail with per-category counts; a card grid of live previews with name, blurb, icon and variant chips; a small footer card (Built in code) explaining that developers register sections through the API and new ones appear automatically.
  • Section hover chrome: inset indigo outline, name chip, and the five-button control toolbar. Move buttons disable at the ends of the list. Delete uses the danger color.
  • Layout variant menu: a small dropdown under Change layout listing variants with a check on the current one. Swapping a variant keeps the content and rerenders instantly.
  • Template picker dialog for new pages, plus the Save page as template dialog (name input, note that marketers and developers will see it under Create a page).
  • Empty states: composer roles see Start with a section; non-composers see No sections yet with copy explaining that a marketer or developer adds sections.
UX notes
  • Role gating: only composer-capable roles (marketer, developer, admin, owner) see insert points and section controls. Content editors can still click into text and edit it inline; reviewers can only comment. The same page renders three different ways depending on who is looking, and the transitions between those states must be invisible, not broken-looking.
  • Live previews in the library are the real components scaled to 34%, so what you see is exactly what you get. This is a differentiator worth protecting; screenshots would drift.
  • Reordering is up/down buttons, not drag and drop. That is a deliberate v1 simplification and it works well on touch, but designers should expect drag to be requested.
  • Duplicate copies the content but not yet the design overrides of the source section, a subtle inconsistency to be aware of.

Known gap: delete has no confirmation. Delete section removes the section immediately with only a success toast (Hero removed) and no undo. An engineering audit flagged the same pattern on page deletes. Designers should spec a confirm dialog or, better, an undo-in-toast pattern, since a section can carry a lot of hand-written copy.

Visual design notes
  • Composition chrome uses indigo as the "editing layer" color, distinct from the brand pink, so editing UI never looks like page content.
  • The insert line and controls appear only on hover with quick opacity transitions; the canvas at rest looks like a clean page.
  • Library cards use a hover lift: indigo border plus a soft indigo shadow, with the add pill fading in centered over the preview.
  • The flash-on-insert ring is the primary "where did it go" affordance; keep it subtle but unmistakable.
Future improvements
  • Drag-and-drop reordering with a drop indicator between sections.
  • Copy a section across pages (a section clipboard).
  • Per-section publish state or locking for regulated content.
  • Registering and editing the section catalog from the /schema builder without code.

Design controls on a section

Token-based design panel

Each section instance carries optional design overrides, edited in a collapsible Design panel inside its card in the form panel. Every value is a named token from the design system (for example paddingY: "lg", theme: "dark", width: "narrow"), never raw CSS. The decision record for this system is DESIGN_CONTROLS.md at the repo root.

Why it exists

Marketers constantly need small design adjustments (more breathing room, a dark hero, a narrower text column) and waiting on a developer for each one kills momentum. Free-form CSS would solve that but destroy visual consistency. Tokens are the middle path: marketers adjust within the design system, so they cannot drift away from it, and the values travel through the headless API as plain data that the customer's frontend maps to its own scale. The philosophy in one line: marketers use the design system, they never get free-form CSS. Developers who need escape hatches have scoped custom CSS on the workbench's Advanced tab.

User journey
  1. A composer-capable user expands the Design disclosure at the bottom of a section card. A small accent dot on the header marks sections that already have overrides.
  2. The panel is grouped like a design inspector: Color (theme, background), Spacing, Layout, Effects and Background image.
  3. Theme is a three-way segment: Inherit / Light / Dark. Choosing Dark flips the whole section to a dark surface with light text, live on the canvas.
  4. Spacing uses three token sliders (Top, Bottom, Sides) that snap across the eight-stop scale 0 / XS / S / M / L / XL / 2XL / 3XL. No pixel input exists anywhere.
  5. Layout offers content width presets (Narrow / Default / Wide / Full), alignment, and a Full viewport height checkbox ("Great for hero sections. Content centers vertically.").
  6. Adding a background image URL reveals an Overlay slider (0 to 100%) that darkens the image behind the text. The slider does not exist until an image is set: controls appear only when they mean something.
  7. A Reset design link clears every override back to the section's coded defaults.
UI components
  • Collapsible Design header with paintbrush icon, customized dot, and chevron.
  • Segmented token pickers, token sliders with stop labels, percentage range rows (section opacity, overlay), checkbox toggles with hints (full height, border top/bottom), corner radius and shadow scales, and a custom color input (swatch plus hex field) when background is set to Custom.
  • On the canvas, overrides paint through a wrapper around the section's own markup; with nothing set the wrapper is a transparent pass-through.
UX notes
  • Developers allow-list which controls each section kind exposes (sectionDesignControls(kind) in the section schema). Every kind gets the standard set; only hero and cta get Full viewport height; chrome kinds (navigation, header, footer) are trimmed to background, theme and typography because they own their own width and padding. Designers should treat the panel as dynamic, not fixed.
  • The panel is marketer-and-up only (the same canCompose gate as composition). In the workbench's section workspace, content editors and reviewers see only the Content and SEO tabs of the five (Content / Layout / Style / SEO / Advanced).
  • Every change previews live on the canvas with no apply step; that immediacy is the whole point of token controls.
  • The dark theme flip must keep text legible automatically (the wrapper applies a light-text scope), and the overlay slider exists specifically so headlines survive busy background photos.
Visual design notes
  • Group titles are small uppercase muted labels; the panel intentionally reads like a familiar design inspector.
  • Selected segments get a subtle primary ring on a raised surface rather than a filled accent, keeping the panel calm.
  • Sliders use the primary accent; stop labels are tabular so they do not jiggle.
Future improvements
  • Wire the spacing scale to per-project brand kit tokens (the storage format already supports it).
  • Element-level design (per-block max width, per-button spacing), the highest-value next step per the decision record.
  • Drag-on-canvas padding handles as interaction polish over the same data.
  • Editing the per-kind allow-list from the /schema builder.

Inline editing and the rich text toolbar

Rich text bubble toolbar

Selecting text on the canvas (in edit mode) raises a floating bubble toolbar above the selection, the same pattern as Notion or Medium. It applies formatting live to the inline-editable text, so the canvas is a real editor, not a preview with an edit dialog.

Why it exists

Inline editing removes the last translation step between "the page" and "the content". Bold, links and heading levels are decisions people make while reading the page in context, so the controls appear exactly there.

User journey
  1. The user clicks into any text on the canvas and selects a range. The bubble appears above (or below when there is no room) the selection.
  2. One-tap buttons cover bold, italic, underline, strikethrough and inline code. A text-color menu offers seven named swatches (Ink, Indigo, Pink, Green, Amber, Red, Slate).
  3. A block-type menu turns the current block into Text, Heading 1 through Heading 6, or Quote, each row previewing its own type style.
  4. The link button opens the link editor: a URL field (placeholder https://your-site.com or /pricing), a Pages button that expands a searchable picker of the project's own pages (title plus path, so internal links never get typo'd), an Open in new tab switch, and a Link relationship single-select radio group (None / nofollow / sponsored / ugc, each with a plain-English hint). Editing an existing link adds a Remove action. Security attributes (noopener/noreferrer) are applied automatically with new-tab links and never shown as options.
  5. The sparkle button opens the AI panel inside the same bubble: a free prompt plus predefined actions (Rewrite, Fix spelling & grammar, Make shorter, Make longer, Simplify language, Make more professional, Make more casual, Translate…) with a back arrow to return to formatting.
UI components
  • The bubble bar itself (horizontally scrollable on narrow screens), the block-type flyout, color flyout, link editor flyout, overflow menu, and the AI panel.
  • Every flyout is portaled to document.body and positioned against its trigger, clamped inside the viewport and flipped above or below when space is tight. This is a hard requirement: the bar can scroll horizontally, so menus rendered inside it would be clipped.
  • Selection preservation: the toolbar snapshots the selection when a menu opens and restores it before applying any command, so clicking inside a portaled panel never loses the highlighted text.
UX notes
  • The same toolbar component is reused across the CMS (entry block editor, markdown surfaces), so its behaviour must stay identical everywhere.
  • The link page picker is the difference between "paste a URL" and "link to a page"; internal links keep working when a path changes only if authors used the picker, which is a good reason to make it the visually dominant option.
  • Turn-into operations replace the block tag but keep the text and inline marks, matching Notion expectations.

Known gap: unsanitised rich HTML. Under the hood, rich inline fields read and write raw HTML (the engineering audit flagged this for sanitisation as a security issue). The designer-visible symptom is that pasted content can carry hidden styling and markup into a field, so text may render with unexpected fonts, colors or structure. Until a sanitiser lands, expect occasional "why does this headline look different" reports, and design a visible "clear formatting" affordance prominently (it already exists in the toolbar's overflow).

Visual design notes
  • The bubble is a compact dark-on-light pill with tight icon buttons; active formats show a pressed state.
  • The AI panel replaces the bar content in place rather than opening a second popover, keeping the interaction anchored to the selection.
  • Rel option chips and page rows reuse the indigo editing-layer accent.
Future improvements
  • Wire the AI actions to the real model (they are currently demo stubs that show the interaction).
  • Keyboard shortcuts surfaced in tooltips (Cmd+B, Cmd+K and friends).
  • Link previews (fetch the target title) and broken-link warnings.

Comments

Field-anchored comment threads

Figma/Sanity-style commenting on page content. Threads anchor to a content field (identified as {sectionId}.{fieldKey}), optionally down to a text range inside that field, so the same thread appears as a pin on the canvas in visual mode and next to the matching input in form mode.

Why it exists

Review happens where the words are. Anchoring threads to fields rather than screen positions means a comment survives layout changes, device switches and mode switches, and reviewers (who cannot edit) still have a first-class way to contribute.

User journey
  1. The user switches the canvas to Comment mode (or stays in form mode and hovers a field).
  2. On the canvas they either click a field's comment affordance or select a run of text; releasing a selection raises a floating Comment button above it. In form mode, hovering a field reveals a small comment button at its top right.
  3. A new-comment popover opens showing the field label and the quoted text, with a composer supporting @mentions (a combobox of teammates) and image attachments.
  4. Existing threads render as circular pins. Clicking a pin opens the thread popover: messages with author, relative time, emoji reactions, reply composer, per-message edit and delete, a resolve toggle, and Copy link (which produces a ?comment={id} URL that reopens the thread on load).
  5. When a thread with a text range is open, the anchored words highlight in the canvas via the CSS Custom Highlight API (a browser feature that paints ranges without altering the markup). In form mode the anchored field gets an indigo ring.
  6. The toolbar Comments button (with an unresolved-count badge) toggles a right-hand panel listing all threads on the page; clicking one scrolls its field into view in whichever mode is active.
UI components
  • Comment pins (avatar-colored circles, active state), the select-to-comment floating button, the new-comment popover, the thread popover, reaction chips, the mention combobox, image attachments in messages, the comments sidebar panel, and count badges in the toolbar and the canvas mode toggle.
  • Pins reposition on every layout change (device switch, section reorder, banner dismiss), driven by a recalculation pass.
UX notes
  • Cross-mode continuity is the signature behaviour: open a thread on the canvas, switch to form mode, and the same thread is anchored to the matching input. Popovers render at the app root so they survive the switch.
  • Reviewers land directly in comment mode and cannot leave it, which makes commenting their whole surface; it must feel complete, not like a restricted mode.
  • Deleting the first message of a thread deletes the whole thread; designers may want a clearer warning for that case.

Known gap: three comment systems coexist. An engineering audit found three separate comment implementations in the codebase: the visual editor's system described here (src/components/cms/editor/comments/CommentSystem.tsx), a newer shared system used by the workbench and entry editor (src/components/cms/comments/, with its own sidebar, selection toolbar and comment-mode provider), and an older layer. Behaviour and visuals can differ between surfaces (for example thread popover styling and what anchors support). Unifying them into one system is a design-debt item; when speccing comment changes, check which surface you are on.

Demo reality. Threads are held in component state with one seeded example thread ("Can we make this punchier?…") so collaboration is visible immediately. They are not persisted across reloads, and mentions do not notify anyone. The interaction model is the deliverable.

Visual design notes
  • Indigo is the comment accent throughout (pins, badges, highlight tint at 28% alpha, field rings), consistent with the editing-layer color.
  • Pins sit slightly above the field's top-right corner with a flattened bottom-left corner, the familiar "speech pin" silhouette.
  • Resolve is a state change, not a delete: resolved threads leave the pin layer but stay listed in the sidebar.
Future improvements
  • One unified comment system across canvas, form, entries and collections (the audit's recommendation).
  • Real persistence and notifications on mention (parts of the comment backend already exist in the demo's infrastructure).
  • Thread filters (open, resolved, mine) and a workspace-level inbox.

Multiplayer presence

Presence signals

A full multiplayer presence layer: who is in the project, which page and section they are on, live named cursors on the canvas, and colored outlines around the section a teammate is editing. It follows the best of the references: Webflow's element outlines, Sanity's field-anchored avatars, and Figma's named cursors, unified by one accent color per person.

Why it exists

Two people editing the same hero without knowing about each other is the classic CMS disaster. Presence gives a constant, ambient answer to "am I about to step on someone", and it also makes the product feel alive during demos and onboarding.

User journey
  1. The top bar shows an overlapping avatar stack of everyone active in the project. Idle peers render faded.
  2. Clicking the stack opens a who-is-here popover grouped into "on this page" and elsewhere, each row showing avatar, name, seat and current location. Every row is a navigation affordance: clicking jumps to that person's page (updating ?page=) and scrolls to their section.
  3. On the canvas, each active peer on the current page shows a named cursor (a colored pointer with a first-name chip) drifting smoothly between positions, plus a colored inset outline and name tag on the section they are editing.
  4. In lists (pages, collection rows) small row-level avatars show who is inside each item, and the entry editor shows Sanity-style field-level avatars beside the exact field a peer is editing.
UI components
  • Avatar stack, rich hover card (photo, name, role, location), grouped popover with jump-to rows, canvas cursor layer, section outline layer, row avatars, field avatars.
  • An eight-color accent palette assigned per person, deliberately distinct from the brand pink so presence never reads as UI accent.
UX notes
  • Cursor motion is tuned to the presence tick: positions interpolate with a CSS transform transition matched to the update interval, so movement reads as gliding rather than teleporting.
  • Role fidelity matters: reviewers never appear to be editing anything, matching what their seat can actually do.
  • Presence layers scroll with the page (they are positioned inside the canvas root, like comment pins), and the cursor layer is hidden on mobile.

Simulated engine over real data. There is no realtime backend in the demo. A presence store plays the role of the websocket: real workspace members from the mock data "join" and move around on a 3.6 second tick, and every location they visit points at real data (actual page paths, section ids, entry ids, field names), so avatars land on rows and fields that exist. It runs only while a surface is subscribed and persists nothing. It was built so designers can evaluate the intended multiplayer feel before the realtime infrastructure exists.

Visual design notes
  • One color unifies every signal for a person: their avatar ring, cursor, outline and name tag.
  • Section outlines are 2px inset rings with a small filled name tag at the top-left corner, matching Webflow's "don't step on me" pattern.
  • Name chips use 10px semibold white-on-color pills; keep them readable but small enough not to obscure content.
Future improvements
  • Real websocket-backed presence (the visual layer is designed to be backend-agnostic).
  • Soft locking or "follow mode" (click a peer to spectate their viewport).
  • Conflict hints when two people edit the same field within seconds of each other.

Compare versions and publishing

Compare versions

A full-screen overlay comparing the page's published snapshot with the current draft, field by field, with word-level diff highlighting and per-field restore.

Why it exists

Before publishing, people want to answer "what exactly am I about to change on the live site". A field-level diff answers it precisely, and per-field restore gives a safe way to back out one change without losing the rest of the draft.

User journey
  1. Once a page has been published at least once, a Compare button appears in the toolbar with an amber badge counting changed fields.
  2. The overlay opens with a header legend: a green Published chip, a swap-arrows icon, and an amber Draft chip. A summary line reads, for example, 3 fields changed · published version captured 2 hours ago.
  3. Each changed field renders as a two-column row (published left, draft right) with word-level highlights: removed words struck through in red on the left, added words highlighted in green on the right. Rows carry Changed, Added or Removed tags, including fields from sections that were deleted from the draft.
  4. Hovering a row reveals Restore published value, which copies the published text back into the draft field and toasts.
  5. Empty states cover both edges: Not published yet and No differences.
UI components
  • Modal overlay (max 1060px), legend chips, per-row field label and section name, diff-highlighted text panes, hover-revealed restore action, empty states.
UX notes
  • Restore is per field, not whole-page rollback; that granularity is the feature. A full "revert draft" action would complement it.
  • The diff strips markup and compares words, so formatting-only changes may not appear; designers should know this boundary.
Future improvements
  • Multiple historical snapshots, not just the latest published one.
  • Side-by-side visual (rendered) compare in addition to text.

Publish menu

One popover, opened from the Publish button, is the single publishing surface for a page. The same component is reused by the Pages list so publishing behaves identically everywhere.

Why it exists

Publishing has several related decisions (where, when, and lifecycle actions) that would otherwise scatter across menus. Collapsing them into one surface makes the mental model simple: everything about "making this live" is behind the one button.

User journey
  1. The header restates the page and its state, including On staging when a draft has been pushed there.
  2. Publish to: two cards, Staging ("Private preview") and Production ("Live site"). The default is smart: already-live pages preselect Production, new drafts preselect Staging.
  3. A private preview link row shows the staging URL with copy and open buttons and the caption Private link. Only workspace members can open it.
  4. For Production, a When choice appears: Publish now or Schedule with a date-time picker.
  5. A summary strip states the outcome, for example Goes live at northwind.com/pricing. The primary button matches the choice: Push to staging, Publish now or Schedule.
  6. A footer row of quiet lifecycle actions: Unpublish (or Unschedule), Save as template (composer roles) and Archive.
  7. Publishing snapshots the page: the current title and sections are captured as the published snapshot with a timestamp. That snapshot is what Compare diffs against, and any later edit flips the state to Unpublished changes.
UX notes
  • Only publish-capable roles see the button at all; there is no disabled-button tease.
  • Escape closes the menu; every action confirms with a toast that names the page and destination.
  • Scheduled pages surface their time in both the status pill area and the menu header.
Future improvements
  • Publish-with-approval when the workflow stages feature (Part covering workflows) gates a page.
  • Batch publish from the Pages list.
  • Timezone display on the scheduler.

Compose, review, publish

Add sections from library→Edit inline + design tokens→Comment + resolve threads→Compare draft vs published→Publish (now / schedule / staging)
Publish→Snapshot captured→Edit again→State: Unpublished changes→Compare shows the diff

Publishing captures the snapshot that Compare later diffs against; the cycle repeats per page.

Where generated pages land

Generator entry points into the editor

The page generators (SEO pages from a keyword list, and the ABM page builder that personalises a page per target account) are launched from the Generate menu in the Content hub at /w/{workspace}/p/{project}/content. Their output is ordinary draft pages in the same pages store the visual editor reads.

Mobile and tablet behaviour

Device tiers in the editor

The editor adapts by viewport tier rather than shrinking uniformly. Tiers are mobile (under 768px, and any short landscape phone viewport under 500px tall), tablet (768 to 1279px) and desktop (1280px and up).

Why it exists

The canvas needs pointer precision and horizontal room; a squeezed canvas on a phone would be worse than no canvas. But content edits (fix a typo, approve a comment) are exactly what people do from phones. So phones get a focused content editor and tablets get the full product, on the theory that iPads often stand in for laptops.

User journey and behaviour
  1. Phone. The visual editor pins itself to form mode; the mode toggle and canvas never render. The user edits section fields in the single centered column, with comment pins and popovers fully working. Presence cursors are skipped. Toolbar labels collapse to icons, and secondary controls hide.
  2. Tablet. The full workbench, compressed: canvas, section composition, design panel, device bar, comments, presence, publish. Developer surfaces stay available.
  3. Desktop. Everything, unchanged.
UX notes
  • Rotating a phone to landscape does not unlock the canvas; short landscape viewports are treated as mobile on purpose.
  • Touch targets in the section control toolbar are 24px squares, small for touch; on tablet, row actions and hover chrome need their touch-visible variants (hover-only affordances must have a visible fallback).
  • The device bar inside the editor (previewing the page at phone width) is unrelated to the app's own tier; testing both together is a classic source of confusion during reviews.
Future improvements
  • A read-only canvas preview on phones (view, comment, but not compose).
  • Larger touch targets and long-press menus for section controls on tablet.

What must stay consistent

  • The canvas always renders light, in both admin themes. It is the visitor's site.
  • Indigo is the editing-layer accent (composition chrome, comments, insert points); brand pink stays reserved for the product UI; presence uses its own per-person palette.
  • One publish surface: any new place that publishes a page must reuse the publish menu, not invent a variant.
  • Design values are tokens end to end. No control may ever accept raw CSS from a marketer.
  • Field identity ({sectionId}.{fieldKey}) is the shared anchor for comments, form inputs, presence and diffs; features that break that key break all four.

Part 8

AI and agents

The agent is BetterCMS's flagship differentiator: a scoped teammate that drafts, audits, and bulk-edits content through the same structured surfaces a person uses. This part covers every AI surface, the trust model that holds them together, and the one principle that can never change: the agent writes drafts and proposals, humans publish.

The trust model

Every AI feature in the product hangs off four rules, defined in src/lib/agent/types.ts and enforced in src/lib/agent/runs-store.ts:

  • Every run plans first, and a person approves the plan before anything executes.
  • Every write is a proposed change the person can accept or reject, one by one or in bulk.
  • Accepted changes apply as drafts through the normal content stores, never straight to production.
  • Nothing publishes. Publishing stays in the publish menus, human only.

This principle must stay. No auto-publish, ever. The two safety rules (plan approval and draft-only writes) are deliberately not configurable, even by workspace admins. The AI controls page states this in the UI: "Agent plans always need a person's approval, and nothing the agent writes publishes on its own." Any future design that weakens this breaks the product's core promise.

One more hard copy rule from AGENT_PLAN.md: users never see a raw model name. Speed tiers are shown only as Lite, Balanced, and Max. The single exception is a user's own bring-your-own-key model, which they chose and named themselves.

Run lifecycle: compose, plan, run, review, apply or discard, undo

Compose prompt→Planning→Plan approval→Applying (streamed steps)→Review changes→Apply selected as drafts→Done
Plan approval→Rejected (nothing ran)
Review changes→Discard (nothing written)
Done→Undo this run (reverts applied changes)

Read-only skills (audit, internal links, AEO) skip the review stage and go straight from applying to done, because they never write.

What is real in the demo. The whole agent system runs on in-browser stores. Steps stream on timers to feel live, proposals are generated deterministically from the project's real pages and collections, and everything resets on a hard reload. The store's own comment says it plainly: in production a backend drives the same status machine over a live connection, and the store's API does not change. Designers should treat the flows, states, and copy as the intended production spec.

The agent dock

Why it exists

Plan mode, runs, and credits

Plan mode and the run thread

Every run is a state machine held in the runs store (runs-store.ts), rendered as a chat-like thread (AgentThread.tsx): the ask, streamed steps, a plan card requiring approval, then staged changes or findings.

Why it exists

Trust comes from showing the work before doing it. The plan is the contract: what the agent will touch, what it will not, and what it will cost, all approved before a single write is staged.

User journey
  1. Send. Status planning: steps stream in (Reading project structure, Following 3 workspace instructions, Checking pages and collections, Preparing a plan).
  2. Status awaiting_approval: the plan card shows the goal, what will be touched, boundary lines (always including the no-publish line), and the credit estimate.
  3. Approve and run moves to applying with skill-specific steps (rename: Reading pages and entries, Finding every mention, Staging the changes). Reject ends the run: Plan rejected. Nothing was changed.
  4. Write skills land in review; read-only skills land in done with findings. The done card summarizes the outcome (Applied 6 changes, Found 4 things worth a look, or Done. Nothing needed changing.) with credits spent on the right.

  • User bubble (right-aligned, surface tint) with the context chips above the prompt text; step list with a spinner on the running step and green checks on finished ones.

Change review, the crown jewel

Multi-document change review

A run can propose changes across many documents at once (a rename might touch fifteen pages and entries). The review UI, Sanity-style, exists in two sizes: an inline card in the thread and dock, and a full-page three-panel review (ChangeReview.tsx) that takes over the Agent page when a run hits review.

Why it exists

Bulk editing is only acceptable when review scales with it. Grouping changes per document, showing a live preview with per-field highlighting, and making everything reversible turns "the agent edited 15 documents" from terrifying into routine.

Three-panel review anatomy

Header: back link, Review changes, N documents
Reasoning: the ask, agent note, boundaries, follow-up box
Proposed changes: per-document cards, Discard, Confirm all
Live document preview with per-field before and after diff

Columns are 300px / 360px / flexible. Under the md breakpoint the reasoning panel hides and the preview stacks below the list.

User journey
  1. The run finishes staging and the page swaps to the review. All proposals start pre-accepted; the header counts affected documents.
  2. The middle panel lists one card per document: a checkbox (the per-document toggle), a kind icon (entry, page, or new entry), the label, and N changes. Clicking a card previews it on the right.
  3. The right panel renders the document as it will look: title, meta pills, then every changed field. Old text shows removed words struck through in red; new text highlights changed words in the primary tint. Brand-new fields carry a green New badge.
  4. The user decides at whatever grain they want: Confirm all, Discard (toast: Changes discarded. Nothing was written.), untick whole documents, or untick single field changes in the inline card view.
  5. With a partial selection, a footer bar appears: 3 of 5 documents selected plus Apply 7 selected. Applying writes the accepted changes as drafts and toasts Changes applied. Saved as drafts. Publishing stays with you.
UI components
  • Left panel: the original prompt in a speech bubble, an optional agent aside (what it deliberately left out, e.g. quoted text during a rename), the plan's boundary lines, and an Ask a follow-up… input that starts a new run.
  • Document cards with tri-purpose rows: select (click), include (checkbox), and status (green check once applied).
  • Field diff blocks: uppercase field label, optional context, before box (muted, strikethrough on removed words), after box (primary border, highlight on new words).
  • Confirm all is a green (emerald) button; Apply N selected is the primary button; Discard is a quiet outline button.
The undo journal
  • Every apply captures a journal of reversible operations (UndoOp in types.ts): remove a created entry or page, or restore a field's previous value on a page, entry, or section.
  • The done card and every applied row in agent history show an instant Undo this run button. Undo replays the journal in reverse, marks the run Reverted, and records an audit event.
  • Undo is honest about conflicts: changes published or hand-edited since are skipped, with the toast {n} kept. Some changes were published or edited since, so they were left alone. The same guard runs at apply time: an accepted proposal whose target was deleted or edited mid-review is skipped rather than force-written.
UX notes
  • Dependent proposals stay consistent automatically: rejecting a "create entry" proposal rejects its field patches; accepting a patch re-accepts its parent create.
  • Unticked items dim instead of disappearing, so the user always sees the full set they are choosing from.
  • Both review sizes call the same store actions, and every apply and undo lands in the project audit trail with the actor name and count.
Visual design notes
  • Diff colors carry the whole grammar: rose for removed, primary tint for changed, emerald for added. Keep these stable; users learn them once.
  • The preview intentionally looks like a document, not a table of patches. That is the Sanity lesson: review reads best in the shape of the thing being changed.
Future improvements
  • Side-by-side full-document diff mode, comments on proposals for team review, editing a proposal's text before accepting, keyboard navigation through documents.

Future improvements
  • Scope editing per grant (read-only keys), last-used timestamps surfaced in the UI, an activity feed of external-agent operations, org-wide keys for CI pipelines, and a first-party skill package that teaches outside agents the project's schema.

What must stay consistent

  • No auto-publish, anywhere. Drafts and proposals from the agent, publishing from humans. Every new AI surface inherits this.
  • Tiers are Lite, Balanced, Max. Never a model name in product UI, except a user's own BYOK model.
  • Estimate before, receipt after. No run starts without a visible credit range; no run ends without the actual spend.
  • Amber means "your move." The two human-gate states (plan approval, change review) always use the amber chip; green is reserved for done and applied.
  • The diff grammar. Rose strikethrough for removed, primary highlight for changed, emerald for added, everywhere changes are reviewed.
  • Everything reversible, everything audited. Applied runs carry an undo journal, and applies, reverts, and grants all land in the audit trail.
  • Governance narrows, never removes. Admin controls can cap and switch off; they can never grant autonomy past the human gates.

Team, roles and editorial workflow

How a team shares one CMS without stepping on each other: who pays for a seat, what each role can see, how work moves from draft to published, and where every access change is recorded. This part covers the seat model, built-in and custom roles, agency guests, the editorial workflow, and the Access page.

The permission hierarchy at a glance

BetterCMS layers access in five tiers. The workspace owner sits on top and is never billed. Below the owner, paid seats do the creating and shipping, free seats do the reading and approving, and guests collaborate from outside the company without touching the bill at all.

Who sits where (and who pays)

  • Workspace owner (no seat, never billed)
    • Paid seats
      • Developer, $20/mo
      • Marketer, $15/mo
      • Content editor, $10/mo
    • Free seats, unlimited
      • Reviewer, $0
      • Viewer, $0
    • Agency guests (no seats, project-scoped)

Guests are whole external teams. They never appear on the bill and never see workspace settings.

Every rule in this part flows from one idea: reading and reviewing content should never cost money or require IT involvement, while creating and publishing are deliberate, priced, permissioned acts.

The seat model

Seats live at the workspace level and are priced by role, not per head at a flat rate. Three things are deliberately free: the owner (always), viewer seats (read-only access), and reviewer seats (view, comment, suggest, approve). The paid seats are Content editor at $10/mo, Marketer at $15/mo, and Developer at $20/mo. The UI celebrates the free tier rather than hiding it, and that celebration must survive any redesign.

Guest teams, the agency model

Agency guests

A host workspace invites a whole external team (an agency or a freelancer group) as guests. Guests hold no seats, cost nothing, are scoped to all projects or a chosen few, and never see confidential workspace settings. The section lives on the Seats and members page under the roster.

Why it exists

Agencies are the default way marketing sites get built. Forcing a client to buy five developer seats for a six-week engagement kills deals, and forcing the agency into one shared login kills accountability. Modeled on Webflow's guest system, this gives the agency its own roster inside the client's workspace, with the agency's admin managing their own teammates.

User journey

Editorial workflow

Every project has an ordered set of workflow stages describing the journey between draft and publish. The defaults are Draft, In review, Changes requested, and Approved, each with its own chip color. Exactly one stage carries the publish gate: publishing is only offered from that stage, so approval stays meaningful. Published itself is not a stage but the system lifecycle beyond the gate.

Stage flow into Published

Draft→In review→Approved (gate)→Published
In review→Changes requested→Draft / In review
Published→Any working stage (starts a new draft version)

Only the gate stage offers Publish. Moving a published entry back to a working stage flips its status to draft and begins a new version.

Plan gating: workflows are a Pro and up feature (below Pro the tab shows a locked state), and customizing the stages themselves is Team and up.

Workflow tab (kanban board and list)

Why it exists

Content teams otherwise track "where is that post?" in a spreadsheet or a Trello board that drifts from the CMS. Here the board is the CMS: stage, assignees, and due dates live on the entry itself, so the board can never be out of date, and the same entry slide-over the Content tab uses opens from a card click.

Why it exists

Comments as a collaboration layer

Comments are the conversation that runs alongside the workflow: field-anchored threads with mentions, reactions, image attachments, replies, and resolve, pinned to the exact field or text range they discuss, in both the visual and the form editor. They are the reviewer seat's primary tool, which is why reviewer seats are free: the whole point is to let anyone say "this sentence is wrong" precisely, without a paid seat. The full editor experience, pins, anchoring, and thread UI, is documented in Part 7; treat comments and workflow as two halves of one review loop, comments for the words and workflow for the state.

Production should unify these: one event stream, filtered per project or workspace.

Approvals philosophy: publishing is the protected act

One principle shows up twice in this product, and naming the consistency matters more than either instance. Publishing is the single protected act. Editing is cheap, commenting is free, drafts are safe to break. The moment content goes live is the moment it can hurt the brand, so that moment is guarded everywhere:

  • In the human workflow, the publish gate: entries publish only from the gate stage, and only seats at marketer rank or above can press the button. Approval is not a ceremony, it is the key.
  • In the agent (Part on the AI agent), the no-auto-publish rule: the agent drafts, renames, generates, and proposes multi-document changes, but a human always confirms, and nothing the agent does can push content live on its own.

These are the same rule wearing two coats. Any future feature that creates or modifies content, human or automated, should inherit it: free to draft, gated to ship. Designers reviewing new flows can use it as a litmus test, and copy should keep teaching it the way the board's error toast already does: Entries publish from the gate stage, so approval stays meaningful.

AI delivery

This tab is new here: it moved from the standalone Markdown manager into SEO so it sits alongside the other machine-consumption surfaces (sitemap, RSS, robots). llms.txt, llms-full.txt and content negotiation

One card with three stacked rows controlling the site's machine-readable delivery. Header copy: Everything on this site, served as markdown for agents and answer engines.

Why it exists

llms.txt is an emerging convention (an H1, a one-line summary blockquote, then H2 sections of links): a markdown index of the site that AI agents fetch first to decide what to read, the way crawlers fetch sitemap.xml. Shipping it as a toggle, generated from real content, means every BetterCMS site is agent-legible with zero effort.

User journey
  1. llms.txt row. Shows the filename in monospace, a mode pill (Auto in neutral gray, Custom in primary tint), and when serving is on, a copyable URL chip ({domain}/llms.txt). Auto mode is described as A markdown index of everything worth reading, generated from the site. Agents and AI dev tools fetch this first. Actions: Preview (renders the generated file, listing published pages with their descriptions, one H2 section per collection, and any standalone markdown files), Edit (a full-height monospace dialog; saving switches the mode to custom, and the footer warns Saving switches llms.txt from auto to your custom version.), Upload (accepts a .txt or .md file and toasts llms.txt replaced with your file), a revert icon back to auto (custom mode only), and a serve switch.
  2. llms-full.txt row. The whole site inlined into one markdown file, every page and entry separated by rules. It has only Preview and a serve switch, and it defaults off. The row copy explains why: The whole site inlined in one file. Large sites can exceed agent context windows, so this stays off unless you need it. (A context window is the maximum text an AI model can read at once.)
  3. Content negotiation row. Not a toggle at all: an Always on emerald check. Every canonical URL answers markdown requests directly, so agents never need a second URL; search engines see one page, agents get clean markdown from the same address. A copyable example makes it concrete: curl -H "Accept: text/markdown" https://{domain}/pricing.
UI components
  • Single bordered card, three rows with icon tiles, mode pill, URL chips with copy icons, ghost action buttons, serve switches, the edit dialog (DialogShell with a 46vh monospace textarea), and the shared markdown preview dialog.
UX notes
  • The auto-versus-custom state machine is the core interaction: auto regenerates forever, custom freezes the file. Every path into custom (edit-save, upload) and back (revert icon, Revert to auto in the dialog) is one click and confirmed by a toast. Never let the user end up in custom mode without noticing.
  • Keeping llms-full.txt off by default is a product opinion, not a limitation; the row copy carries that reasoning and should keep doing so.
  • Preview before enable works: users can read exactly what agents will fetch while the serve switch is still off.
Visual design notes
  • Filenames and URLs are always monospace; the mode pill is the row's only tinted element in auto mode. Emerald is reserved for the always-on check.
  • Rows share the icon-tile plus title plus description plus trailing-actions anatomy used across settings surfaces; match it exactly if rows are added.

  • Fetch analytics: which agents pulled llms.txt and when, closing the loop with the Robots tab's AI crawler allowances.

Media library

Media library

Why it exists

User journey
  1. The user opens Media and sees a three-pane layout: a sidebar (filters, smart views, folders, storage meter, trash), a toolbar (search, view switch, sort), and the asset grid.
  2. They narrow by kind using sidebar filters: Images, GIFs, Videos, Lottie, Documents, Text & Markdown, SVG & icons, Audio, plus Favorites. Assets and their byte sizes are real data and the crop genuinely rewrites the image, but uploads are a staged animation (no file actually leaves the browser), Replace and Download are stubs that toast, format conversion is visual, and asset edits live in component state, so most changes reset on a hard reload.

Media and forms are the two features most often reached from somewhere else, so their shared components carry more consistency weight than their own screens do.

Always the shared MediaPickerDialog. OG images, image blocks and instruction references must browse the same library with the same layout, never a bespoke picker.
Storage math
The library sidebar meter, the inspector's file size, and the Usage page's storage tables all derive from the same per-asset byte counts. If any surface rounds or buckets differently, users will file it as a billing bug.

Monospace value plus a copy icon that flips to a check, used identically for CDN URLs, form endpoints and embed snippets.

The SettingsSection pattern (a hard layout rule)

Every settings group on every page is built from one component, SettingsSection in src/components/cms/SettingsSubNav.tsx. Its defining rule: the section header lives inside the card, not above it. The title and description sit in a header strip with the same horizontal padding (px-5) as the rows below, separated by a hairline border, so the title's left edge aligns exactly with the field labels under it. Do not design floating titles above cards; the audit in WORKSPACE_AUDIT.md explicitly consolidated on this pattern.

Header
Title at 13.5px semibold, optional 12.5px muted description, optional right-aligned action slot (a button, badge, or filter)
Body
px-5 py-4 padding by default; a flush variant drops it so tables sit edge to edge
Rows

Extras

The two-layer pricing model

Every price in the product comes from one file: src/lib/billing/pricing.ts, which its own header calls the single source of truth. The model has two layers that are never merged, plus per-seat pricing on top:

What makes up a workspace bill

1. Workspace plan (the team container)

Free $0→Company $25 / $20 yearly→Agency $38 / $30 yearly

2. Site plan (one per live site)

Free (Starter) $0→Basic $20 / $15→Pro $35 / $25→Team from $1,500/mo→Enterprise custom

3. Paid seats (workspace level)

Viewer free→Reviewer free→Editor $10→Marketer $15→Developer $20

Bill = workspace plan + the sum of every site's plan + paid seats. Yearly billing is the default everywhere, with the headline "Save up to 30% with yearly billing".

Details worth memorising: viewer and reviewer seats are always free and unlimited. AI credits belong to the site plan (100 trial on Free, 1,000 on Basic, 3,000 on Pro, 40,000 on Team), topped up with packs. Team and Enterprise are managed contracts, not self-serve checkouts, and exist as managed workspaces too.

The gating philosophy

Three rules shape every locked or over-limit state, and they are deliberate product positioning, not accidents:

  • Locked features render a calm LockedFeature panel (src/components/cms/billing/FeatureGate.tsx): a lock icon, the feature name, Available on {Plan}, one short blurb, one See plans button, and the reassurance Everything else keeps working. Upgrade only if you need this. No countdowns, no fear.
  • Usage is never red. usageState() returns healthy, approaching (at 80%), or over. Approaching renders amber; over renders sky blue, an informational colour, with the copy Past the included amount. Billed at the add on rate, never cut off.
  • Nothing is ever cut off. The catalog prints the overage promise verbatim: Overage never cuts off mid month. It bills at the add on rate, and a hard cap is available so there is never a surprise bill.

Delivery modes, and what changes per mode

A project's delivery mode decides who renders and serves the site. The model (src/lib/cms/delivery.ts) is two switches, hosting and API, presented as three presets:

ModeWho rendersWhat is onWhat appears in the app
Hosted siteBetterCMS CloudHosting on, API offPage builder and publishing, Domains, SSL and redirects, the Hosting page

  1. Each project renders as a card: project name, domain count, a Manage button, then a small table (Domain, Status, SSL, Added). Primary domains carry a Primary badge; clicking any row routes into that project's Domains settings.
  2. Add domain opens the shared dialog. From the workspace it includes a project picker (Domains are verified and served per project.); from a project page the project is fixed and the picker disappears.
  3. The host input normalises whatever you paste (https://Www.Acme.com/ becomes www.acme.com), validates the hostname shape, and rejects duplicates inline with {host} is already connected.
  4. On add, the workspace page routes you straight into the owning project's Domains settings to finish DNS, keeping the "manage inside the project" rule intact.
  5. An empty workspace shows a dashed empty state: Connect a domain to publish a project under your own brand. Pick the project, add the domain, then follow the DNS steps.
UI components
  • Grouped cards with mini tables, status badges (Active or Verifying), SSL badges (issued, pending, failed), the portal dialog with project select and monospace host input.

Publishing: the tab, the menus, and staging

"Publishing" appears in three places, and they divide cleanly: the Publishing settings tab holds site-wide rules, the page publish menu ships one page, and the entry publish menu ships one collection entry. All three lean on the same concept of staging: a private, always-on copy of the site at {project}.bettercms.site where drafts can be previewed at a real URL before anything goes live. For users, staging is "the safe version of the site only my team can see"; production is the public domain.

Publishing settings tab

Site-wide publishing rules at /w/{ws}/p/{project}/settings/publishing (visible when hosting is on): workflow switches, the environments overview, and build hooks.

Why it exists

Rules that apply to every publish (review requirements, scheduling, auto-deploy) need a home separate from any one page's menu.

User journey
  1. Toggle three workflow switches: Require review (content must be approved before it can be published), Scheduled publishing (allow future-dated publishes), and Auto-deploy on publish (trigger a production deploy whenever a page is published).
  2. Scan the Environments cards (production, staging, preview) with status dot, URL, branch, last deploy time, and Open, Copy, and Deploy buttons.
  3. Copy or regenerate Build hooks: URLs external systems can POST to trigger a build, one each for production and staging.
UI components
  • Three SettingsSection cards, switch rows, environment mini-cards, monospace hook URL rows with copy, regenerate, and delete icons.
UX notes
  • The switches are uncontrolled defaults in the demo (they render checked but persist nothing), and hook actions are visual. The workflow that actually enforces review lives in the stage system (Part 9); production should wire the Require review switch to it.
Visual design notes
  • Environment status reuses StatusDot tones: ready is success, building is warning, failed is danger.
Future improvements
  • Persist the switches and connect them to stage gates.
  • Per-environment publish targets.

Page publish menu

The single publishing surface for a page (src/components/cms/editor/PublishMenu.tsx), used identically from the visual editor header and the Pages list, so a page can be shipped from wherever you meet it.

Why it exists

Publishing decisions (where, when, and whether to preview first) belong in one popover with one mental model, not scattered across toolbars. The component is deliberately self-contained and writes to the pages store directly so every caller behaves the same.

User journey
  1. Open the menu from a page's Publish control. The current state shows as a dot and label: Draft, Published, Unpublished changes, Scheduled, or Archived.
  2. Choose the destination: private Staging or live Production. The default is smart: staging for drafts, production for already-live pages.
  3. Choose the timing: now or later with a datetime input. Scheduling toasts "{title}" scheduled for {date} and flips the state chip to sky-blue Scheduled.
  4. Pushing to staging toasts "{title}" pushed to private staging; publishing now toasts "{title}" is live at {domain}.
  5. Secondary actions round out the lifecycle: copy the private preview link, open the staging or live URL, Save as template, Unpublish (back to draft, with the honest toast It is no longer live.), and Archive.
UI components
  • A 350px popover (or fixed portal when opened from a list row), destination segmented choice, now/later radio with datetime field, state dot and label, action rows with icons, Escape closes.
UX notes
  • The five page states and their colours (draft muted, published emerald, modified amber, scheduled sky, archived faded) are the canonical page lifecycle; the Pages list, editor header, and this menu must always agree.
  • Publishing here is real within the session: state changes flow through the pages store and every surface updates.
Visual design notes
  • One primary button per state; destructive and lifecycle actions are quiet text rows below a hairline. The staging option carries a lock or shield glyph to reinforce "private".
Future improvements
  • Publish notes ("what changed") attached to the action.
  • Bulk publish from the Pages list using the same component.

Entry publish menu and workflow gates

The publish control for a collection entry (EntryPublishMenu.tsx): one tight toolbar cluster with a status pill, a Save button, and a split Publish button whose caret opens publish now, schedule, staging link, and unpublish.

Why it exists

Entries previously showed "Published" in three places; the rework concentrated publishing into exactly one cluster so the word appears once. Entries also have a richer state machine than pages (draft, in review, approved, scheduled, published, archived) because they flow through editorial workflow.

User journey
  1. In the entry editor toolbar: status pill, Save (flashes Saved with a check), then the split Publish button. Once live, the primary label becomes Republish.
  2. The caret menu offers Schedule (datetime dialog, violet Scheduled state, cancellable with Unscheduled), Copy staging link (a tokenised preview URL on staging.bettercms.site), and Unpublish.
  3. Role gating: reviewers see only the status pill. Editors can save but the publish button disables with the tooltip Publishing needs a marketer seat or above.
  4. Stage gating: when a workflow is active, the entry workflow bar drives progress instead: entries in a working stage show Approve to advance toward the publish-gate stage, and only entries sitting at the gate show Publish as the primary action. Workflow stages, assignment, and approvals are covered in Part 9; the contract to remember here is that the gate stage is the only door to production.
UI components
  • Status pill with per-state dot colours (draft zinc, review amber, approved sky, scheduled violet, published emerald), split button, dropdown menu, schedule dialog, toasts.
UX notes
  • The split button is the pattern to reuse anywhere a primary action has secondary timing options. Keep the primary half a verb and the caret a true dropdown, not a second click target for the same action.
Visual design notes
  • The cluster is compact (28px controls) because it lives in a dense toolbar; the publish half is the only filled-primary element in the editor header.
Future improvements
  • Scheduled-publish queue view per project.
  • Publish with references check ("this entry links two drafts").

What must stay consistent

  • One staging story. {project}.bettercms.site is the staging domain everywhere: the Hosting environments card, the Domains preview list, the publish menus, and the agent's staging-only writes all point at the same concept.
  • The status vocabularies are fixed. Deploys: queued, installing, building, deploying, live, failed, cancelled, superseded. Pages: draft, published, unpublished changes, scheduled, archived. Entries add review and approved. Do not invent synonyms.
  • Switching anything reassures. Delivery mode, hosting mode, and downgrade dialogs all end with an explicit "your content does not move" line. Keep it.
  • Publishing is gated by people, not by the UI alone. Role checks (marketer and above) and workflow gate stages both block the action at the store level; new publish surfaces must call the same checks.

UX audit, gaps and the road ahead

This closing part is the designer's work queue. It names what the product already does well, lists every known gap with the evidence behind it, audits each major screen, and ends with the twelve design projects we believe give the most return for the effort. Everything here was verified against the actual code, not remembered from meetings.

How this audit was run

Three inputs. First, the engineering audit that preceded the production handoff plan, which traced every store and route and flagged where the UI and the data disagree. Second, the workspace settings audit in WORKSPACE_AUDIT.md at the repo root, which catalogued every workspace surface with a Keep, Fixed, Roadmap or Skip verdict. Third, a fresh pass for this document through the route files in src/routes and the component tree in src/components/cms, confirming each claimed gap at its exact source location before it was allowed into the tables below. Nothing in this part is secondhand.

What already works well

An honest audit starts with what should not be redesigned. These are the load-bearing patterns of BetterCMS. Changing any of them needs a stronger reason than taste, because each one is applied consistently across roughly eighty screens and users have learned them.

One token system, everywhere

Every screen draws from the same color, spacing and type tokens. There is no rogue screen with its own grays. This is why the product feels like one product, and it is also why dark mode could be added late without a rewrite: the Graphite dark theme remaps the same tokens instead of restyling components one by one.

A unified folder tree

Pages and collections live in one nested tree with real URLs, up to four levels deep. Designers should treat this as the spine of the content model: the same tree drives the sidebar, the Pages hub, breadcrumbs and the Move-to-folder menus. One mental model, many surfaces.

Calm plan gating

Locked features never shout. They render as quiet locked cards with an explanation and an upgrade path, and usage meters never turn red (the highest state is an amber "approaching limit"). The product asks for money without threatening the user. Keep this temperament in anything new.

The agent reviews before it applies

Every AI action produces a plan the user approves, then a change review with per-document Confirm, Discard and Apply-selected, with a live before and after preview. Nothing auto-publishes. This review-before-apply model is the product's core trust story and must survive every future agent feature.

Consistent list machinery

Pages, Markdown files and collection tables share one ListToolbar (search plus contextual filters) and one Paginator (50, 100, 200 per page). A user who learns one list has learned them all.

Copy discipline

Sentence case headings, no dashes in UI copy, a real glossary in COPY_GUIDE.md, and agent speed tiers that are only ever called Lite, Balanced and Max. The copy system is a design asset. New screens should read like the old ones wrote them.

Two smaller wins worth protecting: one-time reveal dialogs for API keys and webhook secrets deliberately block the Done button until the value is copied (a small forcing function that prevents a real support ticket), and disabled switches across settings carry explanatory hints instead of going silently gray.

The seven that deserve expanded notes

Dead menu items are worse than missing menu items. The dashboard project menu mixes fully working actions (Share, Duplicate, Move to folder, Transfer) with three items that have no click handler at all. A user who clicks Archive and sees nothing happen learns to distrust every menu in the product. Until Archive and Delete are actually designed (an archived state needs a home, a filter and a restore path, and Delete needs the confirmation pattern below), the honest fix is to remove or visibly disable them. Never ship a control that swallows a click.

Destructive actions have no house pattern. Deleting a section in the visual editor is instant. Deleting a page asks nothing in some paths. Meanwhile revoking an API key does confirm, and one-time reveal dialogs are carefully engineered. The product knows how to be careful; it just is not careful everywhere. The pattern to design once and apply everywhere: low-cost deletions get an undo toast (five seconds, then commit), high-cost deletions get a typed confirmation, and everything in between gets a confirm dialog naming exactly what disappears.

The comment split is invisible until it bites. There is a field-anchored comment system with pins, reactions, mentions and resolve on the canvas and in form mode, and a separate older implementation inside the entry editor, plus a presence-adjacent layer. Each renders threads with slightly different anatomy. A reviewer who lives in comments all day will hit the seams: a reaction available in one place and not another, a resolve control in a different corner. Part of the fix is engineering (one store), but the thread anatomy, pin behavior and resolve flow need one canonical spec first. That spec is a design deliverable.

The Searchable toggle is a false affordance. In the schema builder, a developer can mark a field as searchable. The search index ignores it; what actually controls indexing is the separate per-field switch inside the Search hub's own configuration. Two switches claim the same job and only one works. Designers should decide which surface owns this decision (our recommendation: the Search hub, with the schema builder showing a read-only mirror that links there) rather than just wiring both.

Share links encode their secret in the URL itself. A share token contains the project id inside the token string, so any browser can decode and open it without asking a server. This is why revocation is impossible in the current model: there is no server keeping a list of valid tokens. The future design needs server-issued links, and those make a real management surface possible: a list of active links with created date, view counts, expiry, optional password, and a working Revoke button. Design that surface now so engineering builds toward it.

View-as needs a scope decision, not just a fix. The view-as control (the top-bar menu that lets a higher role preview the app as a lower one) stores a single global selection. The effective role is then clamped per workspace: you can never view as a role above your actual seat in that workspace. The consequence is quietly confusing. If you are an admin in one workspace previewing as a reviewer and you switch to a workspace where you are only an editor, the preview silently changes meaning, and nothing on screen says so. The design question is whether view-as should be per workspace (remembered separately for each), or should visibly announce itself with a persistent banner naming the current preview role, or both. Our recommendation is both: a per-workspace selection plus an always-visible "Viewing as Reviewer" bar with an exit control, which is also how the safest role-preview systems in other products behave.

Analytics should say what it is. The analytics screen is one of the most polished surfaces in the product: date range picker with comparison mode, traffic series, device, browser and country breakdowns, keyword rows. All of it draws from a generator seeded by the project slug, so the same project always shows the same plausible curves and no number ever corresponds to a real page or entry. As a spec for the future analytics feature it is excellent. As a demo surface it is slightly dishonest, because nothing on screen marks the data as sample data. A small "Sample data" pill in the header would cost nothing and would spare every stakeholder demo the moment where someone asks why traffic did not move after publishing.

Testing note for designers. Because most data is in-browser, a hard reload resets nearly everything to seed state. Folders, share links, projects created through the wizard, workspace name edits and account preferences survive (they are stored in localStorage, the browser's small key-value store). Entry edits, workflow moves, comments and agent runs do not. If a flow you were testing "lost" its state, this is why, not a bug you found.

Accessibility priorities

The foundations are better than most prototypes: new dialogs focus their first input, switches carry aria-labels (the text a screen reader announces for a control), and icon buttons on rebuilt pages are labeled. But coverage is uneven, and the gaps cluster in six places.

  • Keyboard reach of the tree and the kanban. The folder tree and the workflow board are the two most structural surfaces, and both are mouse-first. The tree needs arrow-key navigation (up and down between rows, left and right to collapse and expand) and the board needs a pick-up-and-move model: select a card, press a key to lift it, arrows to choose a column, Enter to drop. Both are solved patterns; they need to be specified for our components.
  • Focus management in portaled layers. Dialogs, flyout menus and the rich-text bubble toolbar all render in portals (they are attached to the end of the page rather than inside their parent, so they can float above everything). Portals break the natural Tab order. The rule to enforce: focus moves into the layer when it opens, stays trapped inside while open, and returns to the triggering control when it closes. Rebuilt settings dialogs already do this; flyouts and the bubble toolbar need an audit.
  • Color is carrying status alone in places. The dashboard marks archived projects with a colored dot, and several tables lean on pill color to distinguish states. Every status needs a text label or icon shape alongside its color. The pills already include text; the bare dots are the offenders.
  • Touch targets on mobile rows. The responsive pass made row actions touch-visible instead of hover-only, which was the right call. The remaining work is size: row action buttons hover around 28 pixels and the minimum comfortable touch target is 44. On mobile the actions can grow even if desktop stays dense.
  • Reduced motion is real but partial. The account preference genuinely disables animation where it is wired, but coverage is not complete: deploy progress animations, agent run shimmer and canvas transitions need to respect it too. Define motion tokens with a reduced variant so future animation inherits the behavior for free.
  • Icon-only buttons need a labeling sweep. Rebuilt screens label their icon buttons; older ones are inconsistent. This is a mechanical audit: every icon-only control gets an aria-label matching its tooltip, and every control with a tooltip gets a tooltip delay consistent with the rest of the app.

Where to start. The tree and the board affect the most users per hour of work. Everything else on this list can be fixed screen by screen; those two need a designed interaction model first.

One inherited good habit to preserve: the settings rebuild established that every new dialog gets a portal at a consistent layer, closes on backdrop click and Escape, carries an aria-label, and focuses its first input on open. Treat that as the minimum bar for any dialog designed from here on, and write it into the component spec rather than trusting each feature team to remember it.

What does not exist yet

So no designer ever assumes otherwise: the following have no implementation at all, not even a simulated one. Each is a future design surface, and each line says what designing it would actually involve.

  • Real deploys. The hosting screen is timers and staged states; designing the real thing means failure taxonomies, retry affordances, rollback confirmation and what a half-failed deploy looks like at 2am.
  • Versioning and branching beyond snapshots. Publish snapshots and the Compare overlay exist, but there is no branch model; designing one means naming, merging, conflict presentation and how a non-technical editor understands "you are on a draft branch".
  • Localization and multi-language. Nothing in the model speaks locale; designing it touches every list (locale switcher), every field (translated or shared), workflow (per-locale publish state) and the folder tree itself.
  • A plugin or extension system. No third-party surface exists; designing one means an extension gallery, permission consent screens, and rules for where foreign UI is allowed to appear inside our shell.
  • A template marketplace. Cloneable template links exist for single projects, but there is no browsable marketplace; designing it means listing pages, quality signals, previews and an install flow that respects plan gating.
  • A real MCP server backend. The Connected agents pages show endpoints and setup commands, but no server answers them; designing the real thing means key scoping UI, per-agent activity logs and revocation flows (MCP is the protocol that lets outside AI tools talk to the CMS).
  • Real billing charges. Checkout is test mode and card update is a toast; designing production billing means real payment forms, failed-payment recovery emails and states, proration explanations and invoice disputes.
  • Email notifications. The in-app notification center works, but nothing sends email; designing it means digest versus instant preferences, email templates that match the brand, and unsubscribe surfaces that meet legal requirements.

The next twelve design bets

Ranked by return on effort: how much user trust or capability each project buys per week of design work. The first three are corrections, the middle six are completions, the last three are foundations.

Two sequencing notes before the list. Bets one through four should land before any new feature work, because each one repairs a place where the product currently breaks a promise, and broken promises compound: every week they ship, more users learn to hesitate before clicking. And bets ten through twelve pay off slowly but cheaply; they are the ones to hand to a designer with partial availability, because each can progress an hour at a time without blocking anyone.

  1. A house pattern for destructive actions. One spec covering undo toasts, confirm dialogs and typed confirmations, then applied to section delete, page delete, the dashboard Delete item and everything else that removes work. This single pattern retires three high-severity findings at once.
  2. Unify the comment system. One thread anatomy, one pin behavior, one resolve flow across canvas, form mode and entry editor. Spec first, migration second.
  3. Share link management. Design the server-issued model: a links table with views, expiry, optional password and a Revoke that actually works. This turns a trust liability into a feature.
  4. Resolve the Searchable split. Decide the owning surface for per-field search configuration, mirror it read-only elsewhere, and kill the false affordance in the schema builder.
  5. Real-data states for large sites. Design what 10,000 pages looks like: virtualized lists (rendering only the visible rows so scrolling stays fast), search-first navigation when trees get deep, and skeleton states that do not lie about progress.
  6. An empty state coverage sweep. Every list, table and panel gets a designed empty state with one primary action, replacing whatever blank space or lorem-adjacent placeholder currently sits there.
  7. Notification preferences. A real preferences page (channel, event type, digest cadence) that persists, replacing the cosmetic toggles, designed together with the future email surface so the two never diverge.
  8. Mobile editor improvements. The mobile tier is deliberately a focused editor; make it excellent rather than reduced: bigger touch targets on rows, a thumb-reachable block menu and a comment flow that works one-handed.
  9. Onboarding per role. The onboarding flow asks who you are and then everyone lands in the same place. Design a first-session path for each seat: a reviewer should land on the board, a developer on the schema, a marketer on the page templates.
  10. Command palette expansion. The field palette in the schema builder proves the pattern; grow it into a product-wide palette for navigation, creation and actions, which also quietly fixes part of the keyboard-access story.
  11. A design system documentation site. The tokens and component conventions live in Markdown files; a browsable, searchable reference with live examples is what keeps the "one token system" strength true as the team grows.
  12. Localization groundwork. Not the full feature: the groundwork. Audit every screen for text expansion (German is roughly thirty percent longer), define where a locale switcher would live, and decide the model for translated versus shared fields, so the eventual feature is an implementation and not an archaeology project.

A closing word. Most of what this audit found are seams, not cracks. The product's core patterns are sound and consistently applied; the debt lives at the edges where features shipped fast and the connective tissue (confirmations, persistence, one comment system) never got its own project. That connective tissue is exactly what this design team is for.
